## Open â€” large scope (multi-round work)

### #4 + #5: Postgres-specific field types + RLS â€” remaining Phase 4

Phases 1 + 2 + 2.5 + 3 + 4.0 shipped: DbPool enum, backend-aware connect, boot-time mismatch detection, backend-aware DDL rendering, per-variant apply loop, backend-agnostic QuerySet terminals, Postgres inspectdb, and the portable `Json` field. Each remaining piece is independent and lands incrementally:

- **Phase 4.1 â€” ArrayField.** Postgres-only `Vec<T>` column with the `text[]` / `int[]` family of types. Needs `SqlType::Array(Box<SqlType>)` or an inner-type encoded variant. System check via `BackendFeature::ArrayColumns` so SQLite use fails at boot.
- **Phase 4.2 â€” JSON operators.** Path access (`->`, `->>`), containment (`@>`, `<@`) for Postgres. Cross-backend story: SQLite gets a JSON1-extension fallback or the operators stay PG-gated.
- **Phase 4.3 â€” full-text search.** `tsvector` / `to_tsquery` + a `FullTextField` column. Postgres-only behind `BackendFeature::FullTextSearch`.
- **Phase 4.4 â€” CIDR / INET / MACADDR.** Postgres network address types.
- **Phase 4.5 â€” RLS plugin.** Third-party `umbra-rls` plugin crate via the M7 contract. `Plugin::on_ready` runs `ALTER TABLE ... ENABLE ROW LEVEL SECURITY` and installs policies.

Each piece is its own session: too much for autonomous ship-and-verify in one round, and a half-finished refactor with broken tests is worse than no refactor.

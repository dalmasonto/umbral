Project roadmap, June 2026

The path to a boringly productive v0.1.

Umbral is moving from a working framework loop into the hardening pass: fewer sharp edges, stronger defaults, better plugin contracts, and the operations pieces real apps need before launch.

Milestones

What shipped, what is being hardened, and what waits behind it.

The roadmap is grouped by outcome rather than date. Work moves forward when the capability is useful enough to dogfood in this site and boring enough to document.

01 Released

The app loop

Umbral can already build and run a real website with model-backed content, admin tools, uploads, auth, and API surfaces.

  • Model derive, QuerySet, relations, migrations, soft deletes, and signals.
  • Auto admin with CRUD, search, filters, relation pickers, and dashboard widgets.
  • REST, OpenAPI, playground, auth, sessions, OAuth, storage, and security middleware.
02 Toward v0.1

Production hardening

The current work is about completing the everyday rails a deployed application expects around the already-shipped core.

  • Email sending through SMTP and API backends, plus a hardened background task queue.
  • Testing and factory helpers for models, handlers, forms, fixtures, and pages.
  • Caching, rate limiting, structured logging, and metrics that can be enabled as plugins.
03 Planned

Ecosystem depth

After the v0.1 baseline, the focus shifts to better realtime surfaces, richer APIs, and stronger community plugin workflows.

  • WebSockets and SSE for user-targeted and room-targeted realtime updates.
  • Nested writable serializers, CSV and Excel import-export, and deeper API docs.
  • Plugin review workflows, compatibility signals, security reports, and directory trust data.

Workstreams

The roadmap is split across four tracks.

Framework core

ORM, migrations, routing, forms, templates, storage, security, and the plugin lifecycle.

Developer experience

Testing helpers, factories, fixtures, examples, docs, project templates, and clearer errors.

Operations

Tasks, email, caching, rate limiting, logging, metrics, health checks, and realtime push.

Plugin ecosystem

Prebuilt plugins, community directory, review data, security reports, and compatibility signals.

Roadmap rules

The project stays useful by refusing fake certainty. A capability graduates when it is dogfooded, documented, and replaceable through the plugin contract.

  • Dogfood first: this website should prove framework features before the page claims them.
  • Plugins are the boundary: official batteries do not get a private path through the framework.
  • Status is honest: planned work is marked planned, missing numbers stay missing, and shipped means documented enough to use.

Follow the work where it happens.

The public roadmap summarizes direction. The changelog, feature catalog, and GitHub repository carry the details as work ships.

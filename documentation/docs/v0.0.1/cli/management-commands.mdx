Every management subcommand runs **against your project's own
binary**. Your `main.rs` hosts the commands via
[`umbra_cli::dispatch`](#how-it-works); `cargo run -- <command>` is
how you invoke them.

This is the Django `manage.py` shape: a project-local script that
imports project code, so the commands see your models and your
plugins. The framework-global [`umbra`](./startproject) binary handles
scaffolding (`startproject`, `startapp`) and nothing else — it can't
manage your database because it doesn't know about your models.

## How it works

Your `src/main.rs` (the one `umbra startproject` generates) ends
with:

```rust
let app = App::builder()
    .settings(settings)
    .database("default", pool)
    .model::<Article>()
    .plugin(blog::BlogPlugin::default())
    .build()?;

umbra_cli::dispatch(app).await
```

`dispatch(app)` parses `argv`, picks the subcommand, and runs it
against the already-built App. The pool, model registry, and plugin
list are all published by `App::build` before the subcommand fires,
so every command sees the same state your server would.

Run them with `cargo run --`:

```bash
cargo run -- serve         # boot the HTTP server
cargo run -- migrate       # apply pending migrations
cargo run -- makemigrations  # write a new migration file
cargo run -- showmigrations  # list per-plugin state
```

## Settings come from the environment

Settings load the same way the server does — defaults, an optional
`umbra.toml`, and `UMBRA_`-prefixed environment variables (last wins).

```bash
export UMBRA_SECRET_KEY=$(openssl rand -hex 32)
export UMBRA_DATABASE_URL=sqlite://app.db?mode=rwc

cargo run -- makemigrations
cargo run -- migrate
```

<Callout type="warning">
Without `UMBRA_DATABASE_URL` set, the management subcommands fall back
to the `database_url` in `umbra.toml` (or `sqlite::memory:` if that's
also unset). `migrate` will happily report `Applied 1 migration(s)`
against an in-memory pool that vanishes when the process exits. Set
the URL once in `umbra.toml` or your shell, and forget about it.
</Callout>

## serve flags

```bash
cargo run -- serve [--addr 0.0.0.0:3000]
```

| Flag | Effect |
|---|---|
| `--addr <host:port>` | Override `settings.bind_addr`. Useful for `0.0.0.0` to listen on all interfaces. Equivalent to setting `UMBRA_BIND_ADDR`. |

## inspectdb flags

```bash
cargo run -- inspectdb \
    --output plugins/imported \
    [--mark-applied]
```

| Flag | Effect |
|---|---|
| `--output <path>` | Where `models.rs` and `migrations/app/0001_initial.json` land. Required. |
| `--mark-applied` | Record `0001_initial` in `umbra_migrations` without running the SQL. Pass when the target DB already holds the introspected tables. |

See [the inspectdb guide](/v0.0.1/migrations/inspectdb) for the full
workflow.

## Errors

Every subcommand prints errors through the `Display` impl, not the
`Debug` repr. A failure reads as a single human-readable diagnostic:

```
error: umbra inspectdb: column `doc.payload` has unsupported SQL type
       `BLOB`; add a matching SqlType variant or edit the generated
       model by hand
```

Exit code is `1` on any error, `0` on success.

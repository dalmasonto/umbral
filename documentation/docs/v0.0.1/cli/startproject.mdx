## Three names, what each one is

Three Cargo entities share the umbra name. Worth pinning down before
the install commands below:

| Name | Kind | Where you see it | What it is |
|------|------|------------------|-----------|
| `umbra` | Library crate (framework) | `Cargo.toml` dep, `use umbra::prelude::*` | The framework itself - `App::builder`, ORM, web, plugins. |
| `umbra-cli` | Library + binary crate | `Cargo.toml` dep + `cargo install umbra-cli` | The CLI tooling. The **library half** exposes `umbra_cli::dispatch(app)` that your `main.rs` calls so `cargo run -- <cmd>` works. The **binary half** is what gets installed globally as `umbra` (next row). |
| `umbra` | Executable on PATH | After `cargo install umbra-cli` | The scaffolding tool - `umbra startproject` / `umbra startapp`. Yes, the binary is called `umbra` but its crate is `umbra-cli`; same Cargo convention as `cargo` (crate name) vs `cargo-something` (subcommand). |

So a real user touches umbra in **two distinct places**:

```bash
# 1. ONCE globally - installs the `umbra` scaffolding binary on PATH:
cargo install umbra-cli
```

```toml
# 2. Every project - both crates as deps in `Cargo.toml`
#    (the project that `umbra startproject` generates already lists them):
[dependencies]
umbra = "..."        # the framework library
umbra-cli = "..."    # for `umbra_cli::dispatch(app)` in main.rs
```

There's no separate `cargo install umbra` step - the `umbra` *binary*
comes from the `umbra-cli` *crate*. The `umbra` *crate* is a library
you only ever depend on, never install.

## What the global `umbra` binary does

It does three things: `startproject`, `startapp`, and `startplugin`. Everything else (`serve`, `migrate`, `createsuperuser`, …) runs against your project's own binary via [`cargo run -- <command>`](./management-commands).

<Callout type="info">
**Two CLIs, one mental model.** The global `umbra` binary scaffolds
projects and plugins - it doesn't need a project to run because it
*creates* one. Every other command (`serve`, `migrate`,
`makemigrations`, `inspectdb`, `dumpdata`, `loaddata`,
`createsuperuser`, `tasks-worker`, and any custom subcommand a
plugin contributes) runs **inside your project** via
`cargo run -- <command>`. Your binary hosts those by ending
`main()` with `umbra_cli::dispatch(app).await`.

```text
GLOBAL  (cargo install umbra-cli; nothing else needed)
  umbra startproject <name>
  umbra startapp <name>
  umbra startplugin <name>

PROJECT (inside a project, after umbra startproject)
  cargo run -- serve
  cargo run -- migrate / makemigrations / showmigrations
  cargo run -- inspectdb / dumpdata / loaddata
  cargo run -- createsuperuser            (from umbra-auth)
  cargo run -- tasks-worker               (from umbra-tasks)
  cargo run -- <your-custom-command>      (from any plugin you write)
```
</Callout>

## startproject

```bash
umbra startproject myblog
cd myblog
cargo run -- migrate   # apply auth, sessions, and Post migrations
cargo run -- serve     # boot at http://127.0.0.1:8000
```

What you get is a **complete blog-style demo** - not a hello-world skeleton.
The first `cargo run -- serve` boots into a working application that exercises
every major umbra surface.

```
myblog/
├── Cargo.toml          # umbra + all built-in plugins + tokio
├── src/main.rs         # ~110 lines wiring every surface (see tour below)
├── umbra.toml          # framework settings
├── .env                # working dev env file (do not commit)
├── .env.example        # env-variable cheat sheet
├── .gitignore
├── README.md           # what's in the project, how to run
└── templates/
    ├── base.html       # Tailwind CDN, nav bar, {% block content %}
    ├── home.html       # home page with post count
    ├── dashboard.html  # login-gated view with post list
    ├── 404.html        # rendered on path miss
    └── 500.html        # rendered on handler panic
```

### What the scaffold demonstrates

| Surface | Where |
|---|---|
| `Post` model with `ForeignKey<AuthUser>` | `struct Post` in `main.rs` |
| Migrations auto-run on boot | `auto_migrate()` in `main.rs` |
| Public home route | `GET /` → `home()` handler + `home.html` |
| JSON API endpoint | `GET /api/posts` → `api_list_posts()` |
| `login_required_html("/login")` layer | `/dashboard` route in the router |
| `LoggedIn<AuthUser>` extractor | `dashboard()` handler signature |
| `umbra::transaction` | inside `dashboard()` wrapping the ORM fetch |
| `AuthPlugin` + `SessionsPlugin` | `.plugin(...)` in the builder |
| `RestPlugin` with a `post` resource | `.plugin(RestPlugin::default().resource(...))` |
| `AdminPlugin` | `.plugin(AdminPlugin::default())` |
| `OpenApiPlugin` (Swagger UI at `/openapi/`) | `.plugin(OpenApiPlugin::new())` |
| Custom 404 / 500 templates | `.not_found_template("404.html")` + `.server_error_template("500.html")` |
| Tailwind classes (CDN) | `templates/base.html` |

Flags:

| Flag | Effect |
|---|---|
| `--path <dir>` | Parent directory the new project goes under. Defaults to `.`. |
| `--local <path>` | Path-dep every umbra crate against a local repo checkout instead of the public `git = "..."` URL. For framework contributors iterating without a remote push. `startapp` and `startplugin` accept the same flag. |

The command refuses to overwrite an existing directory - pick a
different name or move the old one aside.

## Tour of the scaffold

### src/main.rs

The generated `main.rs` is intentionally readable in one pass. It is structured
in three sections: **models**, **app wiring**, and **handlers**.

**Models section** declares a `Post` struct with a `ForeignKey<AuthUser>` author
field. The `#[derive(Model)]` macro generates the `Model` trait impl, the ORM
manager, and the typed column constants (`post::ID`, `post::TITLE`,
`post::PUBLISHED`, `post::AUTHOR`).

**App wiring** calls `App::builder()` with every plugin registered in order: `AuthPlugin`, `SessionsPlugin`, `AdminPlugin`, `RestPlugin`, `OpenApiPlugin`, and `SecurityPlugin`. The `RestPlugin` call chains `.resource(ResourceConfig::new("post"))`, which exposes JSON CRUD at the `/api/post/` endpoint. Query-string filtering is on by default, so `?published=true` and `?title__icontains=rust` work out of the box; call `.disable_filters()` on the resource to turn it off.

**Handlers** show three patterns: a public HTML page (`home`), a plain JSON
response (`api_list_posts`), and a login-gated view (`dashboard`) that uses both
the `LoggedIn<AuthUser>` extractor and `umbra::transaction` to wrap a DB fetch
atomically.

### templates/

`base.html` loads Tailwind via CDN so the generated pages look reasonable without
a build step. Replace the CDN tag with a compiled stylesheet for production.
`home.html` and `dashboard.html` demonstrate template inheritance (`{% extends
"base.html" %}`), context variables (`{{ post_count }}`), and Jinja-style loops.

### .env vs umbra.toml

`umbra.toml` holds configuration that is safe to commit (bind address, dev environment flag, dev secret key). `.env` holds the same values but is listed in `.gitignore` - it is the working override file you never check in. `Settings::from_env()` reads `.env` automatically, and real process environment variables still win over duplicate keys from the file. In production, set `UMBRA_SECRET_KEY`, `UMBRA_DATABASE_URL`, and `UMBRA_ENVIRONMENT=prod` in your environment directly.

## startapp

A plugin in umbra is what Django calls an app. `startapp` scaffolds a
new plugin crate at `plugins/<name>/`:

```bash
umbra startapp posts
```

Generates:

```
plugins/posts/
├── Cargo.toml          # umbra dep
└── src/
    ├── lib.rs          # PostsPlugin + Plugin trait stub
    └── models.rs       # an empty place to declare your first model
```

The generated `lib.rs` looks like:

```rust
pub mod models;

use umbra::plugin::{AppContext, Plugin, PluginError};
use umbra::web::Router;

#[derive(Debug, Default, Clone)]
pub struct PostsPlugin;

impl Plugin for PostsPlugin {
    fn name(&self) -> &'static str {
        "posts"
    }

    fn models(&self) -> Vec<umbra::migrate::ModelMeta> {
        // Register every model the plugin owns so makemigrations
        // picks them up. Uncomment + extend once you've defined one
        // in src/models.rs.
        // vec![umbra::migrate::ModelMeta::for_::<models::Example>()]
        Vec::new()
    }

    fn routes(&self) -> Router {
        Router::new()
    }

    fn on_ready(&self, _ctx: &AppContext) -> Result<(), PluginError> {
        Ok(())
    }
}
```

After scaffolding, the command prints two follow-up edits you make
yourself:

1. Add the plugin crate as a dependency in your project's `Cargo.toml`:

   ```toml
   posts = { path = "plugins/posts" }
   ```

2. Wire it into your `App::builder()` chain in `src/main.rs`:

   ```rust
   .plugin(posts::PostsPlugin::default())
   ```

We don't edit those files automatically - Cargo manifests and your
`main.rs` can be in any shape, and silently rewriting them is more
surprise than help. The two lines you add are explicit and easy to
undo.

<Callout type="info">
Plugin names follow the same rules as Cargo crate names - ASCII
alphanumeric, underscore, hyphen, can't start with a digit. The
struct name is `pascal_case(name) + "Plugin"`, so `blog-engine`
becomes `BlogEnginePlugin`.
</Callout>

### Names umbra refuses to scaffold

`startapp` (and `startplugin` below) reject two classes of names up front so you don't get a half-broken plugin you have to delete and recreate:

1. **A plugin with the same name already exists** at `plugins/<name>/`. Move it aside or pick a different name.
2. **The name collides with a built-in umbra plugin.** Scaffolding refuses these so you can never end up trying to register both `.plugin(my_auth::AuthPlugin)` and `.plugin(umbra_auth::AuthPlugin)` in the same builder chain (the route mounts and migration table names would collide at boot). Reserved names: `admin`, `app`, `auth`, `cache`, `email`, `openapi`, `permissions`, `rest`, `rls`, `security`, `sessions`, `signals`, `static`, `tasks`.

Pick a name that names your domain (`blog`, `billing`, `inventory`) rather than the capability the built-in already provides.

## startplugin

`startapp` is the minimal stub. `startplugin` is the richer version - same destination (`plugins/<name>/`), but the generated scaffold is shaped for a plugin you intend to publish or share across projects:

```bash
umbra startplugin widgets
```

Generates:

```
plugins/widgets/
├── Cargo.toml              # real deps: umbra, serde, sqlx, chrono, async-trait
├── README.md               # file-structure tour + wiring instructions
└── src/
    ├── lib.rs              # WidgetsPlugin with the full Plugin impl
    ├── models.rs           # one example #[derive(Model)] with attributes
    └── handlers.rs         # one example axum handler
```

The generated `src/lib.rs` is the full Plugin impl, not the stub - `models()` returns the example model's meta so the migration engine picks it up, `routes()` registers the example handler at `/<name>/hello`, `on_ready()` is wired and ready for setup work.

The generated `src/models.rs` shows the field-type attributes most plugin authors hit on day one:

```rust
#[derive(Debug, Clone, sqlx::FromRow, Serialize, Deserialize, umbra::orm::Model)]
pub struct WidgetsItem {
    pub id: i64,

    #[umbra(string, max_length = 200)]
    pub title: String,

    pub status: WidgetsStatus,           // a #[derive(sqlx::Type)] choice enum

    #[umbra(noedit)]
    pub published_at: Option<DateTime<Utc>>,
}
```

The generated `src/handlers.rs` shows how to read query params and return JSON:

```rust
pub async fn hello(Query(params): Query<HelloParams>) -> Json<HelloResponse> {
    let who = params.name.as_deref().unwrap_or("widgets");
    Json(HelloResponse { greeting: format!("Hello, {who}!") })
}
```

After scaffolding, the printed next-steps walk you through wiring it into your project: add the path dep in `Cargo.toml`, register the plugin in `App::builder()`, then `cargo run -- makemigrations && cargo run -- migrate` to apply the example model's schema.

### When to pick which

| Choosing... | Use |
|---|---|
| An internal module that just needs a `Plugin` seam | `startapp` (one `lib.rs`, easy to read in one sitting) |
| A reusable plugin you'll share or publish | `startplugin` (separate `models.rs` / `handlers.rs`, README, real deps) |
| You're learning the Plugin trait surface | `startplugin` - seeing the full impl is more useful than a stub |

Same reserved-name list and same already-exists guard for both.

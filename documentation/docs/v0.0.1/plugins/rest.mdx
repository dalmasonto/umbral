Register `RestPlugin` and every model your app declares gets a full JSON CRUD surface at
`/api/<table>/` — no serializer code, no view functions, no router wiring. The cost is
zero control over the response shape in v1; the endpoints serialize the whole struct as-is.

## Minimal example

Add `umbra-rest` to your `Cargo.toml`, then pass the plugin to the builder:

```rust
use umbra::prelude::*;

#[derive(Debug, Clone, serde::Serialize, serde::Deserialize, sqlx::FromRow, Model)]
pub struct Article {
    pub id: i64,
    pub title: String,
    pub body: String,
    pub published_at: Option<chrono::DateTime<chrono::Utc>>,
}

#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error>> {
    let settings = Settings::from_env()?;
    let pool = umbra::db::connect(&settings.database_url).await?;

    App::builder()
        .settings(settings)
        .database("default", pool)
        .model::<Article>()
        .plugin(umbra_rest::RestPlugin::default())
        .build()?
        .serve("0.0.0.0:3000".parse()?)
        .await?;

    Ok(())
}
```

`RestPlugin::default()` is equivalent to `RestPlugin::new()`. Both register the same routes.

## What gets generated

For each exposed model the plugin mounts five endpoints under `/api/<table>/`:

| Method | Path | Behaviour |
|---|---|---|
| `GET` | `/api/<table>/` | List all rows. Returns `{"results": [...], "count": N}`. |
| `POST` | `/api/<table>/` | Create a row. Returns `201` + the inserted row. |
| `GET` | `/api/<table>/{id}` | Retrieve one row by primary key. `404` on miss. |
| `PUT` / `PATCH` | `/api/<table>/{id}` | Update a row. Both methods update only the fields present in the request body. Returns `200` + the updated row. |
| `DELETE` | `/api/<table>/{id}` | Destroy a row. Returns `204`. `404` if the row doesn't exist. |

The table name in the URL is the snake-case table name the migration engine uses — the same
value as `#[umbra(table = "...")]` if you set one explicitly, otherwise the snake-case of the
struct name.

Errors come back as `{"error": "...", "code": "..."}` with an appropriate HTTP status. The
error code vocabulary (`not_found`, `bad_input`, `database_error`, `invalid_json`) is not
yet stable.

## Exposing only some models

By default every registered model is exposed except the three internal tables
(`auth_user`, `session`, `umbra_migrations`). Those are always blocked regardless of what
you configure.

**Allowlist — only these tables:**

```rust
.plugin(
    umbra_rest::RestPlugin::new()
        .include_only(["article", "tag"])
)
```

**Blocklist — everything except these tables:**

```rust
.plugin(
    umbra_rest::RestPlugin::new()
        .exclude(["audit_log", "payment"])
)
```

The two methods are mutually exclusive in effect: `include_only` wins if both are called
because any table not on the allowlist is hidden, so the extra `exclude` list is redundant.
Use one or the other.

## What's not in v1

<Callout type="info">
The four items below are known follow-ons tracked in `bugs/gaps.md`. Each has a workable
workaround today.

- **Custom serializer fields.** Endpoints serialize the whole struct as-is. There is no
  `get_<field>(obj)` hook to reshape or hide individual fields. Workaround: write a
  dedicated handler for that endpoint and register it on your own router before adding
  `RestPlugin` (axum matches the first route that fits).
- **Per-field exclusion** (e.g. hiding `password_hash` from a `User` response). Workaround:
  keep sensitive columns in a separate model and call `.exclude()` on it.
- **Auth gating.** Every exposed route is open in v1. Wrap the app router with a Tower
  middleware layer or an axum extractor to enforce authentication before requests reach the
  REST handlers.
- **Pagination.** List endpoints return all rows in the table. High-row-count tables will
  return everything in one response. Cursor or page-number pagination is a planned follow-on.
</Callout>

## Pairing with OpenAPI

`RestPlugin`'s URLs are exactly what `umbra-openapi` documents — register both plugins and
you get a live Swagger UI over the generated endpoints at no extra cost. See
[OpenAPI plugin](./openapi) for setup.

Register `RestPlugin` and every model your app declares gets a full JSON CRUD surface at
`/api/<table>/` - no serializer code, no view functions, no router wiring. The cost is
zero control over the response shape in v1; the endpoints serialize the whole struct as-is.

## Minimal example

Add `umbra-rest` to your `Cargo.toml`, then pass the plugin to the builder:

```rust
use umbra::prelude::*;

#[derive(Debug, Clone, serde::Serialize, serde::Deserialize, sqlx::FromRow, Model)]
pub struct Article {
    pub id: i64,
    pub title: String,
    pub body: String,
    pub published_at: Option<chrono::DateTime<chrono::Utc>>,
}

#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error>> {
    let settings = Settings::from_env()?;
    let pool = umbra::db::connect(&settings.database_url).await?;

    App::builder()
        .settings(settings)
        .database("default", pool)
        .model::<Article>()
        .plugin(umbra_rest::RestPlugin::default())
        .build()?
        .serve("0.0.0.0:3000".parse()?)
        .await?;

    Ok(())
}
```

`RestPlugin::default()` is equivalent to `RestPlugin::new()`. Both register the same routes.

## What gets generated

For each exposed model the plugin mounts five endpoints under `/api/<table>/`:

| Method | Path | Behaviour |
|---|---|---|
| `GET` | `/api/<table>/` | List all rows. Returns `{"results": [...], "count": N}`. |
| `POST` | `/api/<table>/` | Create a row. Returns `201` + the inserted row. |
| `GET` | `/api/<table>/{id}` | Retrieve one row by primary key. `404` on miss. |
| `PUT` / `PATCH` | `/api/<table>/{id}` | Update a row. Both methods update only the fields present in the request body. Returns `200` + the updated row. |
| `DELETE` | `/api/<table>/{id}` | Destroy a row. Returns `204`. `404` if the row doesn't exist. |

The table name in the URL is the snake-case table name the migration engine uses - the same
value as `#[umbra(table = "...")]` if you set one explicitly, otherwise the snake-case of the
struct name.

Errors come back as `{"error": "...", "code": "..."}` with an appropriate HTTP status. The
error code vocabulary (`not_found`, `bad_input`, `database_error`, `invalid_json`) is not
yet stable.

## Exposing only some models

By default every registered model is exposed except the three internal tables
(`auth_user`, `session`, `umbra_migrations`). Those are always blocked regardless of what
you configure.

**Allowlist - only these tables:**

```rust
.plugin(
    umbra_rest::RestPlugin::new()
        .include_only(["article", "tag"])
)
```

**Blocklist - everything except these tables:**

```rust
.plugin(
    umbra_rest::RestPlugin::new()
        .exclude(["audit_log", "payment"])
)
```

The two methods are mutually exclusive in effect: `include_only` wins if both are called
because any table not on the allowlist is hidden, so the extra `exclude` list is redundant.
Use one or the other.

## Customising the response shape

Three builder methods let you reshape what the REST endpoints emit on a per-table,
per-field basis. They apply to every outbound response (`list`, `retrieve`, `create`,
`update`/`patch`) in order: **hide → transform → computed**.

### Bundling per-model in a `ResourceConfig`

When the customisation grows past a couple of lines, move it next to the model in a
`ResourceConfig`. The `RestPlugin::default().resource(config)` builder folds it in;
the per-call `.hide` / `.transform` / `.computed` builders below still work for
one-off cases.

```rust
// In your plugin / app: plugins/users/src/lib.rs (or src/users.rs)
use serde_json::json;
use umbra_rest::ResourceConfig;

pub fn rest_resource() -> ResourceConfig {
    ResourceConfig::new("user")
        .hide("password_hash")
        .transform("email", |v| {
            let s = v.as_str().unwrap_or("");
            match s.split_once('@') {
                Some((_, d)) => json!(format!("***@{d}")),
                None => v.clone(),
            }
        })
        .computed("display_name", |row| {
            let f = row.get("first_name").and_then(|v| v.as_str()).unwrap_or("");
            let l = row.get("last_name").and_then(|v| v.as_str()).unwrap_or("");
            json!(format!("{f} {l}").trim().to_string())
        })
}

// In main.rs: one line per app.
let rest = RestPlugin::default()
    .resource(users::rest_resource())
    .resource(posts::rest_resource());
```

The DRF analogy: each Django app has a `serializers.py` next to its `models.py`.
`ResourceConfig` plays the same role - per-model REST customisation lives next to
the model, not in the project's wiring layer.

Multiple `.resource(...)` calls for the same table stack additively. Mixing
`.resource(config)` with one-off `.hide(table, field)` calls in `main.rs` is fine -
everything lands in the same per-table vecs.

### Per-call builders (one-off cases)

```rust
use serde_json::json;

let rest = RestPlugin::default()
    .hide("user", "password_hash")
    .transform("user", "email", |v| {
        let s = v.as_str().unwrap_or("");
        match s.split_once('@') {
            Some((_, d)) => json!(format!("***@{d}")),
            None => v.clone(),
        }
    })
    .computed("user", "display_name", |row| {
        let f = row.get("first_name").and_then(|v| v.as_str()).unwrap_or("");
        let l = row.get("last_name").and_then(|v| v.as_str()).unwrap_or("");
        json!(format!("{f} {l}").trim().to_string())
    });
```

| Builder | Effect |
|---------|--------|
| `.hide(table, field)` | Strip the field from every response. The column stays writable via POST/PUT/PATCH and readable through the ORM - only the outbound JSON is shaped. |
| `.transform(table, field, fn)` | Replace a field's value. The closure receives the raw `serde_json::Value`, returns the replacement. The key stays the same; the value changes. |
| `.computed(table, name, fn)` | Add a derived field. The closure receives the (already-customised) row map and returns the value for the new key. No underlying column needed - the field exists only at the API. |

The order is intentional. Hide first so the row doesn't carry secrets into the
transform / computed closures. Computed runs last so it sees the masked /
stripped row, not the raw one.

<Callout type="info">
**On inputs.** Hide / transform / computed only shape **outbound** responses. POST,
PUT, and PATCH still accept and write the underlying columns as-is. To restrict
what clients can write, gate the endpoint with auth middleware or hand-roll the
handler.
</Callout>

## Pagination

`RestPlugin` exposes a `Pagination` trait so list endpoints can be wrapped in any
envelope shape. Three built-ins ship, plus a custom-trait escape hatch - the same
shape DRF uses.

### Built-ins

```rust
use umbra_rest::{LimitOffsetPagination, NoPagination, PageNumberPagination, RestPlugin};

// Default - no pagination. Envelope: `{ results, count }` where count == results.len().
// No LIMIT applied; no extra COUNT query runs.
RestPlugin::default();

// Django shape: `?page=N&page_size=M`.
// Envelope: `{ count, total_pages, current_page, page_size, next, previous, results }`.
RestPlugin::default()
    .paginate(PageNumberPagination::new(20).with_max_page_size(100));

// REST classic: `?limit=N&offset=M`.
// Envelope: `{ count, limit, offset, next, previous, results }`.
RestPlugin::default()
    .paginate(LimitOffsetPagination::new(50));
```

For `PageNumberPagination`:

| Query param | Default | Effect |
|-------------|---------|--------|
| `?page=N` | `1` | Page number. Invalid values fall back to `1`. |
| `?page_size=M` | `page_size` config | Rows per page. Clamped to `max_page_size`. Disable client overrides via `.lock_page_size()`. |

`next` and `previous` in the envelope are integer page numbers (not URLs) so the
shape stays host-agnostic. Clients combine the path they already know with the
returned page number to fetch the next/previous page.

### Custom envelope

Implement `Pagination` directly. The two methods are `extract_request` (URL params →
limit / offset) and `paginate` (rows + total → JSON envelope):

```rust
use std::collections::HashMap;
use serde_json::{json, Map, Value};
use umbra_rest::{PageRequest, Pagination};

struct WindowPagination { window: u64 }

impl Pagination for WindowPagination {
    fn extract_request(&self, params: &HashMap<String, String>) -> PageRequest {
        let cursor = params.get("cursor").and_then(|s| s.parse().ok()).unwrap_or(0);
        PageRequest { limit: self.window, offset: cursor, page: None }
    }

    fn paginate(&self, rows: Vec<Map<String, Value>>, total: i64, req: &PageRequest) -> Value {
        let next_cursor = req.offset + req.limit;
        json!({
            "data": rows,
            "meta": {
                "total": total,
                "next_cursor": (next_cursor as i64 < total).then_some(next_cursor),
            },
        })
    }
}

RestPlugin::default().paginate(WindowPagination { window: 25 });
```

`extract_request` should be forgiving - return defaults on parse failure rather than
raising errors, so a client typo on a query param doesn't 400 every list request.

<Callout type="info">
**Don't return URLs in `next` / `previous`.** Returning page numbers / cursors / offsets
keeps the envelope identical whether the API is served from `localhost`, behind a
reverse proxy, or via CDN. The client knows the path; let it build the URL.
</Callout>

## Authentication

`RestPlugin::authenticate(impl Authentication)` wires an auth backend that runs on
every request. The backend inspects request headers and returns
`Option<Identity>` - `Some(...)` when a caller is recognised, `None` when the
request is anonymous. Permissions (next section) decide what that identity is
allowed to do.

The trait is intentionally narrow: header-only inspection, async (because most
backends hit the DB to verify credentials), no error variants (silent `None` on
invalid credentials so failure modes don't leak info to the client).

### Built-ins

```rust
use umbra_rest::{Authentication, ChainAuthentication, FnAuthentication, Identity,
                 NoAuthentication, RestPlugin, parse_basic_credentials};

// Default - every request is anonymous.
RestPlugin::default();

// Session-cookie auth (via umbra-sessions, the recommended browser path):
RestPlugin::default().authenticate(FnAuthentication::new(|headers| async move {
    let user = umbra_sessions::current_user(&headers).await.ok().flatten()?;
    Some(Identity::user(user.id).with_staff(user.is_staff))
}));

// HTTP Basic Auth (via umbra-auth, the curl path):
RestPlugin::default().authenticate(FnAuthentication::new(|headers| async move {
    let (user, pass) = parse_basic_credentials(&headers)?;
    let auth_user = umbra_auth::authenticate(&user, &pass).await.ok()?;
    Some(Identity::user(auth_user.id).with_staff(auth_user.is_staff))
}));

// Try session first, fall back to Basic Auth:
use std::sync::Arc;
let auth = ChainAuthentication::new(vec![
    Arc::new(session_auth) as Arc<dyn Authentication>,
    Arc::new(basic_auth)   as Arc<dyn Authentication>,
]);
RestPlugin::default().authenticate(auth);
```

The plugin doesn't bundle a session-cookie or Basic-Auth helper because most
apps use exactly one; bundling all of them would force unused transitive deps
on every user. The 5-line `FnAuthentication` closures above are the canonical
shape.

JWT, API key, mTLS, OAuth - same pattern. Wrap the verifier in
`FnAuthentication::new(...)`, return `Some(Identity)` on success.

## Permissions

Permissions sit on the `ResourceConfig` so they live next to the model:

```rust
use umbra_rest::{IsAuthenticated, IsStaff, ReadOnly, ResourceConfig};

// Public read, no write at all.
ResourceConfig::new("article").permission(ReadOnly)

// Every action requires a logged-in user.
ResourceConfig::new("comment").permission(IsAuthenticated)

// Staff-only.
ResourceConfig::new("audit_log").permission(IsStaff)
```

### Built-ins

| Permission | Anonymous | Authenticated (non-staff) | Staff |
|-----------|-----------|--------------------------|-------|
| `AllowAny` (default) | ✓ everything | ✓ everything | ✓ everything |
| `IsAuthenticated` | 401 | ✓ everything | ✓ everything |
| `IsStaff` | 401 | 403 | ✓ everything |
| `ReadOnly` | ✓ List/Retrieve, 403 writes | ✓ List/Retrieve, 403 writes | ✓ List/Retrieve, 403 writes |

### Combinators

`AndPermission::new(vec![Box::new(P1), Box::new(P2)])` requires every check.
`OrPermission` allows when any check passes (short-circuits on first success).
The right shape for "staff OR owner" rules:

```rust
use umbra_rest::{IsStaff, OrPermission};

struct IsOwner;
impl umbra_rest::Permission for IsOwner {
    fn check(&self, _action: umbra_rest::Action, id: Option<&umbra_rest::Identity>)
        -> Result<(), umbra_rest::PermissionError>
    {
        // Custom check against id.user_id + the requested row's owner column.
        # Ok(())
    }
}

ResourceConfig::new("post").permission(OrPermission::new(vec![
    Box::new(IsStaff),
    Box::new(IsOwner),
]))
```

### Custom permission classes

Implement the trait directly. The signature is sync (`fn check(&self, Action,
Option<&Identity>) -> Result<(), PermissionError>`) because permission checks
walk in-memory rule sets - they don't hit the DB. If you need DB-driven checks
(per-row ownership), do them in the route handler or in a `transform` after
the row is loaded.

## Opt-in views

By default every model exposes all five actions (`List` / `Retrieve` /
`Create` / `Update` / `Delete`). For resources where some methods don't make
sense, scope the views explicitly:

```rust
use umbra_rest::{Action, ResourceConfig};

// Catalogue: list + retrieve only. Create/Update/Delete return 404
// - the endpoint isn't conceptually mounted.
ResourceConfig::new("product")
    .views([Action::List, Action::Retrieve])

// Append-only audit log.
ResourceConfig::new("audit_event")
    .views([Action::List, Action::Retrieve, Action::Create])
```

The unexposed-action body is `{"error": "action `Delete` is not exposed on
`/api/product/`", "code": "not_found"}` - a 404 with an explanatory message,
which is the right semantic (the resource exists, but that operation isn't
part of its API).

## Custom actions (`@action`)

CRUD covers most resources but not all. Django REST Framework hangs the not-CRUD endpoints off the same viewset via `@action`:

```python
class PostViewSet(ViewSet):
    @action(detail=True, methods=['post'])
    def publish(self, request, pk=None):
        ...

    @action(detail=False, methods=['get'])
    def recent(self, request):
        ...
```

The umbra-rest shape is a builder call on `ResourceConfig`:

```rust
use http::Method;
use serde_json::json;
use umbra_rest::{ActionError, ActionScope, ResourceConfig};

ResourceConfig::new("post")
    // POST /api/post/{id}/publish/ - one row at a time.
    .action("publish", Method::POST, ActionScope::Detail, |ctx| async move {
        let id: i64 = ctx.pk.as_deref().unwrap_or("0").parse()
            .map_err(|_| ActionError::BadInput("bad id".into()))?;
        // Hit the ORM; flip a flag; return whatever JSON the client needs.
        Post::objects().filter(post::ID.eq(id))
            .update_values(serde_json::Map::from_iter([
                ("published".into(), json!(true)),
            ]))
            .await
            .map_err(ActionError::internal)?;
        Ok(json!({ "id": id, "published": true }))
    })
    // GET /api/post/recent/ - collection-scope.
    .action("recent", Method::GET, ActionScope::Collection, |ctx| async move {
        let limit: i64 = ctx.query.get("limit").and_then(|s| s.parse().ok()).unwrap_or(10);
        let rows = Post::objects()
            .order_by(post::CREATED_AT.desc())
            .limit(limit as u64)
            .fetch()
            .await
            .map_err(ActionError::internal)?;
        Ok(json!({ "results": rows }))
    })
```

| Scope | URL shape | Has `ctx.pk`? |
|-------|-----------|---------------|
| `Collection` | `/api/<table>/<name>/` | No |
| `Detail` | `/api/<table>/<id>/<name>/` | Yes - the raw URL segment |

Both with and without the trailing slash are accepted.

**The handler signature.** Any async closure of `Fn(ActionContext) -> impl Future<Output = Result<Value, ActionError>>`. The `ActionContext` carries:

- `table`, `name` - the URL segments.
- `pk` - `Some(string)` for detail-scope, `None` for collection-scope. Parse with `.parse::<i64>()` (or whatever your PK type is).
- `identity` - the same `Option<Identity>` `Authentication` produces, after permission has approved the call.
- `body` - the parsed JSON body, or `Value::Null` when the request had none.
- `query` - `HashMap<String, String>` of the query string.

**The error variants** (`ActionError`) map onto the same JSON envelope the CRUD handlers use: `BadInput` → 400, `NotFound` → 404, `Unauthenticated` → 401, `Forbidden` → 403, `Internal` → 500.

**Permissions.** Custom actions run through the same `Permission::check` the built-in actions do, with `Action::Custom("publish")` (or whatever the name is). A resource gated by `IsAuthenticated` rejects anonymous calls to `/publish/` just like it rejects anonymous `POST /` to create. A permission that wants to special-case a specific action can match on the variant:

```rust
struct PublishersOnly;
impl Permission for PublishersOnly {
    fn check(&self, action: &Action, identity: Option<&Identity>) -> Result<(), PermissionError> {
        // Only authenticated callers can publish; reads are open.
        if action.is_read() { return Ok(()); }
        if action.is_custom("publish") {
            return identity.map(|_| ()).ok_or(PermissionError::Unauthenticated);
        }
        Err(PermissionError::Forbidden)
    }
}
```

`Action::Custom(_)` is NOT subject to `.views([...])` - custom actions opt in by being registered, the view scope only filters the five built-in CRUD actions.

## What's not in v1

The REST plugin is feature-complete for v1. The roadmap items below are
follow-ons rather than gaps:

- **Per-row permissions in the SQL filter.** `IsOwner`-style rules currently
  run in user code after the row is loaded. A future hook lets a permission
  inject a WHERE clause so the database does the filtering and never returns
  unauthorised rows.
- **`OPTIONS` introspection.** Currently REST doesn't reply to `OPTIONS
  /api/<table>/` with a permission summary; DRF's "OPTIONS prints the schema"
  pattern lands when there's a concrete consumer.

## Pairing with OpenAPI

`RestPlugin`'s URLs are exactly what `umbra-openapi` documents - register both plugins and
you get a live Swagger UI over the generated endpoints at no extra cost. See
[OpenAPI plugin](./openapi) for setup.

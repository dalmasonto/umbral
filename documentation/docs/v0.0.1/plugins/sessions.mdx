`umbra-sessions` gives you a DB-backed session table, a UUID cookie, and helpers
for creating, reading, and destroying sessions - enough to implement login/logout
without any additional infrastructure.

## What you get from `SessionsPlugin::default()`

Registering the plugin creates one migration: a `session` table with columns `id`
(UUID string, the cookie value), `user_id` (nullable FK to `auth_user`), `data`
(JSON string), `created_at`, and `expires_at`. The plugin has no hard dependency
on `umbra-auth`; anonymous sessions are valid. The `umbra_session` cookie is set
with `Secure`, `HttpOnly`, `SameSite=Lax`, and `Max-Age=1209600` (14 days,
matching Django's `SESSION_COOKIE_AGE` default).

```rust
App::builder()
    .plugin(AuthPlugin::default())
    .plugin(SessionsPlugin::default())
    .build()?;
```

## Reading the current user in a handler

The one-call helper `current_user` parses the cookie header, queries the session
table (deleting the row lazily if expired), and hydrates the `AuthUser`. Use it
as a guard at the top of any authenticated handler:

```rust
use umbra_sessions::current_user;
use umbra::web::{HeaderMap, Html, StatusCode};

async fn dashboard(headers: HeaderMap) -> Result<Html<String>, StatusCode> {
    let Some(user) = current_user(&headers).await.map_err(|_| StatusCode::INTERNAL_SERVER_ERROR)? else {
        return Err(StatusCode::UNAUTHORIZED);
    };
    Ok(Html(format!("<h1>Hello, {}</h1>", user.username)))
}
```

## Setting and reading per-session data

`get_data` and `set_data` round-trip typed values through the `data` JSON column.
Use them for flash messages, shopping-cart state, or any short-lived server-side
storage that doesn't need its own table:

```rust
// On login: create session, persist data, write the Set-Cookie header.
let session_id = umbra_sessions::create_session(user.id, None).await?;
umbra_sessions::set_data(&session_id, "cart_count", &0_u32).await?;
let cookie = umbra_sessions::set_cookie_header(&session_id, None);

// On a later request: read the data back.
let session = umbra_sessions::read_session(&session_id).await?.unwrap();
let count: Option<u32> = umbra_sessions::get_data(&session, "cart_count")?;
```

## What is NOT in v1

- **Cookie-only signed store.** All sessions are DB rows. A stateless signed-cookie
  alternative is deferred until a real low-traffic use case asks for it.
- **Session middleware.** Handlers call `current_user(&headers)` explicitly.
  `Plugin::middleware()` is the M7 deferral; the explicit extractor pattern is
  intentional - auth checks are visible in the handler signature rather than
  implicit.
- **Automatic cleanup.** Expired rows are deleted lazily on read. A `clearsessions`
  management command or a `umbra-tasks` periodic job lands in a later milestone.

The spec lives in `docs/specs/02-plugin-contract.md`. Source: `plugins/umbra-sessions/src/lib.rs`.

# Static Files

`umbra-static` serves a directory of assets at a URL prefix, wrapping `tower_http::services::ServeDir`. Mount it on the app builder and every file under the directory is reachable at the prefix:

```rust
App::builder()
    .plugin(StaticPlugin::new("/static", "./assets"))
    .build()
    .await?;
```

`ServeDir` handles MIME sniffing, range requests, `If-Modified-Since`, and ETags out of the box.

## Cache headers

By default no `Cache-Control` header is added and browsers apply their own heuristics. Call `.max_age(duration)` to add `Cache-Control: public, max-age=<seconds>` on every response:

```rust
use std::time::Duration;

StaticPlugin::new("/static", "./assets")
    .max_age(Duration::from_secs(86400)) // 1 day
```

### Dev-mode opt-out

When `settings.environment == Dev`, the effective `max-age` is forced to `0` regardless of the configured value. Browsers re-validate every asset on every request in development, preventing stale CSS/JS from masking changes.

```rust
// In production this sends Cache-Control: public, max-age=604800 (1 week).
// In development it sends Cache-Control: public, max-age=0.
StaticPlugin::new("/static", "./assets")
    .max_age(Duration::from_secs(60 * 60 * 24 * 7))
```

The dev-mode check reads the ambient settings at route-build time (when `Plugin::routes` is called), not at constructor time.

## Referencing files from templates

The mount path is the URL prefix; reference assets at `<mount>/<relative-path>` and the browser hits the same handler that serves the file. Reaching them from a `minijinja` template is just normal HTML:

```jinja
{# In wrapper.html or base.html #}
<link rel="stylesheet" href="/static/css/app.css"/>
<script src="/static/js/htmx.min.js"></script>
<link rel="icon" href="/static/favicon.svg" type="image/svg+xml"/>
```

```jinja
{# Inside a page template #}
<img src="/static/img/logo.svg" alt="Acme" class="h-8"/>
<a href="/static/docs/eula.pdf">Terms of service</a>
```

Subdirectories under `./assets` map directly. If `./assets/css/app.css` exists on disk, `/static/css/app.css` serves it. The plugin does no rewriting — what you put on disk is what the browser sees.

### A small template helper

If your asset paths change between dev and prod (different mount point, cache-busted filenames), thread the mount through the template context once and reference it from there:

```rust
// In your handler
let ctx = context!(
    static_url => "/static",
    user       => user.username,
);
templates::render("dashboard.html", &ctx)?;
```

```jinja
{# dashboard.html #}
<link rel="stylesheet" href="{{ static_url }}/css/app.css"/>
```

This is the same pattern Django's `{% static "..." %}` tag uses, just hand-rolled. A `static_url` template global may land in v0.1 to make the mount path implicit.

## Production note

Behind a reverse proxy (nginx, Caddy, Cloudflare), serve static files from the proxy and skip this plugin in prod. It exists for development, single-binary deployments, and apps small enough that the ops overhead of a separate file server is not worth it.

Umbra ships with first-class PostgreSQL support. This page covers the three things you need to run an existing umbra app against Postgres: configuring the connection URL, running migrations, and issuing queries.

## Configure

Set `DATABASE_URL` to a `postgres://` (or `postgresql://`) URL in your environment or `umbra.toml`. `umbra::db::connect` dispatches on the URL scheme and returns the matching `DbPool` variant.

```rust
use umbra::prelude::*;
use umbra::db::connect;

#[tokio::main]
async fn main() -> anyhow::Result<()> {
    let settings = Settings::from_env()?;
    let pool = connect("postgres://user:pass@localhost/mydb").await?;

    App::builder()
        .settings(settings)
        .database("default", pool)
        .model::<Article>()
        .build()?
        .serve()
        .await?;

    Ok(())
}
```

At `.build()` time umbra cross-checks `settings.database_url` against the pool's runtime backend. A mismatch (e.g. the URL says `sqlite:` but the pool is `DbPool::Postgres`) is a hard `BuildError::DatabaseBackendMismatch` at boot, not a silent failure at runtime.

## What works today

These three phases shipped together and give you the full Postgres path end-to-end:

- **Phase 1** (`3228f57`) â€” `DbPool` enum, URL-scheme dispatch in `connect`, `AppBuilder::database` accepts `impl Into<DbPool>`, boot-time backend mismatch check. See `crates/umbra-core/src/db.rs`.
- **Phase 2** (`1dfdd97`) â€” The migration engine renders Postgres DDL: `bigserial` for `i64` PKs, native `ALTER COLUMN SET/DROP NOT NULL` instead of the SQLite table-recreation dance. The apply loop branches on the `DbPool` variant and uses `$1..$N` placeholders and `ON CONFLICT DO NOTHING` on Postgres. See `crates/umbra-core/src/migrate.rs`.
- **Phase 2.5** (`6f24fbd`) â€” `Model` is no longer pinned to `SqliteRow`. QuerySet terminals (`fetch`, `first`, `count`, `exists`) dispatch on the `DbPool` variant. `QuerySet::on_pg(&PgPool)` and `Manager::on_pg(&PgPool)` are the explicit-pool overrides, mirroring the existing `.on(&SqlitePool)`. See `crates/umbra-core/src/orm/queryset.rs`.

A model declared with the standard derive just works on both backends:

```rust
#[derive(Debug, Clone, sqlx::FromRow, umbra::orm::Model)]
pub struct Article {
    pub id: i64,
    pub title: String,
    pub body: String,
}
```

`#[derive(sqlx::FromRow)]` emits a generic-over-row-type impl, so no per-backend code is needed in the model itself.

### Running migrations

```bash
# inside crates/
cargo run -p umbra-cli -- migrate
# -> Applied N migration(s)   (Postgres DDL rendered automatically)
```

### Querying

The ambient path uses whatever pool `App::build()` registered:

```rust
let articles = Article::objects().fetch().await?;
```

If you hold a `PgPool` directly (integration tests, scripts outside an `App`):

```rust
let articles = Article::objects().on_pg(&pg_pool).fetch().await?;
```

- **Postgres-specific field types** (Phase 4) â€” `Array`, `HStore`, `Jsonb`, `Cidr`, and other network types are not yet in the `SqlType` catalogue.

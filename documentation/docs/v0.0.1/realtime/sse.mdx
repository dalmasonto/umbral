# Real-time push (SSE)

`umbra-realtime` pushes data from the server to connected clients, to one user or to a named group/room, without you tracking sockets by hand. Two transports share one targeting API: SSE (one-way server to client, proxy-friendly, `EventSource`) and WebSocket (bidirectional). SSE is the default; reach for WebSocket when the client also needs to send.

<Callout type="info">
A single instance needs no configuration. To run several instances behind a load balancer, point them at a shared Redis with `RealtimePlugin::redis(url)` (the `redis` feature) and `to_user(42)` routes across all of them. Same API; see [Scaling across instances](scaling).
</Callout>

## Wiring

```rust
use umbra_realtime::RealtimePlugin;

App::builder()
    .plugin(AuthPlugin::<AuthUser>::default())   // identity at handshake
    .plugin(SessionsPlugin::default())
    .plugin(RealtimePlugin::default())           // mounts /realtime/sse + /realtime/ws
```

## Sending

From anywhere (a handler, a signal, a task), build a target and `send` a named event with a JSON payload:

```rust
use umbra_realtime::Realtime;

Realtime::to_user(42).send("notification", &json!({ "title": "Build passed" })).await;
Realtime::to_group("chat:123").send("message", &msg).await;
Realtime::broadcast().send("ping", &json!({})).await;
```

| Target | Reaches |
|---|---|
| `to_user(uid)` | every live connection authenticated as that user (laptop + phone) |
| `to_group(name)` | every connection that joined that room |
| `broadcast()` | every connection |

`send` is fire-and-forget. It never blocks on socket I/O, so it's safe to call inside a request handler. A connection that can't keep up is skipped (best-effort), never back-pressured. When `RealtimePlugin` isn't installed, `send` is a no-op rather than a panic.

## Connecting (the browser): `umbra.realtime.subscribe`

Connect through the client helper, **not** a raw `EventSource`. Include it once, then subscribe with the rooms you want and a map of event-name handlers:

```html
<script src="/realtime/client.js"></script>
<script>
  umbra.realtime.subscribe("public:plugin-42", {
    note: function (data, raw) {
      // data is the parsed JSON payload; raw.channel is the event's channel,
      // raw.id is the SSE event id
      insertNote(data);
    },
  });
</script>
```

`subscribe(groups, handlers)` returns `{ unsubscribe }`. The connection's user is resolved from the session cookie at handshake, so `to_user(...)` finds it with no extra work.

### One connection per browser — across all tabs *and* all subscriptions

A raw `new EventSource(...)` opens **one connection per tab**. Open the same page in a handful of tabs and you exhaust the browser's ~6-connections-per-host HTTP/1.1 budget — later requests on that origin start hanging as if the server were down.

`umbra.realtime.subscribe` collapses that to **exactly one** server connection per browser, no matter how many tabs are open or what mix of subscriptions they hold. A `SharedWorker` holds a single `EventSource` over the **set-union** of every tab's groups; each event is routed to the tabs interested in its channel. Open unlimited tabs with any mix of rooms and it stays one connection — when a tab subscribes or unsubscribes a room that changes the union, the worker reconnects that one stream (its `Last-Event-ID` fills the brief gap); an unchanged union never reconnects. When the last tab closes, the connection closes.

<Callout type="info">
**Wire format is internal.** On the wire every event ships under a single `u` SSE event type carrying a channel-tagged envelope `{c, e, d}` (channel, event-name, data) — that's what lets one union connection route to the right tabs. You never see it: the `umbra.realtime.subscribe` API above is unchanged, and the helper unwraps the envelope and calls your handler by event name. A group's channel is its own name; a `to_user` event is `@user:{id}` (delivered to all that browser's tabs); a `broadcast` is `@broadcast`.
</Callout>

<Callout type="info">
**Graceful fallback.** On browsers without `SharedWorker` (or when its construction is blocked by CSP), `subscribe` transparently opens a per-tab `EventSource('/realtime/sse?groups=…')`, listens for the single `u` event, unwraps the envelope, and routes by channel to the same handlers — identical behavior, just one connection per tab instead of per browser. If `EventSource` is also missing, `subscribe` is a no-op. You never write the fallback yourself.
</Callout>

## Group access: `GroupPolicy`

Joining a room is gated. By default only `public:*` groups are allowed, so a client can't subscribe to `tenant:99` or `chat:123` it has no claim to (a rejected group fails the handshake with `403`). Grant access from the authenticated identity by supplying a policy:

```rust
struct MyPolicy;
impl umbra_realtime::GroupPolicy for MyPolicy {
    fn can_join(&self, user_id: Option<i64>, group: &str) -> bool {
        if group.starts_with("public:") { return true; }
        match group.strip_prefix("user:") {
            // a user may join their own private channel
            Some(id) => user_id.map(|u| u.to_string()) == Some(id.to_string()),
            None => false, // extend with your membership / tenant checks
        }
    }
}

RealtimePlugin::default().group_policy(MyPolicy)
```

## WebSockets

For two-way traffic (chat, presence, typed commands), connect to `GET /realtime/ws?groups=...` instead. The same `Realtime::to_user`/`to_group`/`broadcast` push API works unchanged; events arrive as JSON text frames `{"event":"…","data":{…}}`. Inbound client frames go to a handler you register:

```rust
use umbra_realtime::{MessageContext, MessageHandler, Realtime};

struct Chat;
#[umbra_realtime::async_trait]
impl MessageHandler for Chat {
    async fn on_message(&self, ctx: &MessageContext, text: String) {
        // ctx.conn_id / ctx.user_id let you authorize + route
        let msg: ChatMsg = serde_json::from_str(&text).unwrap();
        Realtime::to_group(&msg.room).send("message", &msg).await;
    }
}

RealtimePlugin::default().message_handler(Chat)
```

```js
const ws = new WebSocket("ws://localhost:8000/realtime/ws?groups=public:lobby");
ws.onmessage = (e) => { const { event, data } = JSON.parse(e.data); /* … */ };
ws.send(JSON.stringify({ room: "public:lobby", body: "hi" }));
```

The same `GroupPolicy` gate applies at the WS handshake (a denied group fails the upgrade). The default `MessageHandler` ignores inbound frames, which is fine for push-only apps.

## Model fan-out (zero polling)

Push automatically whenever a model changes, the "live dashboard" story, by bridging the ORM's [signals](../orm/signals) to a real-time send. `on_model::<T>()` subscribes to `T`'s `post_save` / `post_delete`; each fire hands you a `ModelEvent` (table, action, the row as JSON, the actor):

```rust
use umbra_realtime::{ModelEvent, Realtime, RealtimePlugin};

RealtimePlugin::default()
    .on_model::<Post>(|ev: ModelEvent| async move {
        // a create/update/delete on Post → everyone watching that post
        Realtime::to_group(format!("post:{}", ev.pk().unwrap_or(0)))
            .send("changed", &ev).await;
    })
    // and the admin notification bell: any new submission → the staff room
    .on_model::<Submission>(|ev| async move {
        Realtime::to_group("staff").send("submitted", &ev).await;
    })
```

`on_table("name", ...)` is the untyped form (when you don't have the model type handy). The handler runs server-side on every write: no client polling, no glue code.

## Reconnect resume (`Last-Event-ID`)

Networks drop. When a browser's `EventSource` loses its connection it reconnects on its own, and on that reconnect it sends back the `id:` of the last event it saw as the `Last-Event-ID` header. `umbra-realtime` keys off that to fill the gap with no missed events.

Every delivered event is stamped with a process-global, strictly-increasing sequence id, written as the SSE `id:` line. The registry keeps a bounded **replay buffer** of recent events. On a reconnect carrying `Last-Event-ID`, the server replays — *before* attaching the live stream — exactly the buffered events newer than that id **that this connection's target would have received** (its user, its joined groups, and broadcasts; never another user's private events). A first connection with no `Last-Event-ID` behaves as before: live only.

It's automatic — the browser side is just `umbra.realtime.subscribe(...)`:

```js
umbra.realtime.subscribe("public:lobby", {
  // On a network blip the connection reconnects and replays missed events for
  // you; you write no resume logic. raw.id is the event's seq.
  message: (data, raw) => render(data),
});
```

Size the buffer with `replay_buffer(n)` (default 1024; `0` disables replay, live-only):

```rust
RealtimePlugin::default().replay_buffer(4096)
```

<Callout type="warning">
**Bounded-buffer caveat.** The buffer holds only the most recent `n` events. If a client is offline long enough that the event it missed has already been evicted, it resumes from the oldest *retained* event and silently skips anything older — there's no infinite history. Size `n` to cover the longest drop you want to bridge at your peak event rate. WebSocket has no native `Last-Event-ID`, so reconnect resume is an SSE feature; a WS client re-subscribes and gets the live stream from the moment it reconnects.

With the Redis backplane, the replay buffer is per-instance (local): a reconnect that lands on a *different* instance than before replays only that instance's recent buffer. For seamless cross-instance resume, pin a client to one instance (sticky sessions) or keep `n` small and treat replay as best-effort.
</Callout>

## Connection limits (`max_connections`)

Cap the total number of live connections across **both** transports so a runaway client (or a load test) can't exhaust file descriptors:

```rust
RealtimePlugin::default().max_connections(10_000)
```

The default is unlimited. Once the cap is reached, a new SSE or WebSocket handshake is refused with `503 Service Unavailable` instead of opening the stream; a disconnect frees a slot and the next connection is admitted immediately. `EventSource` treats the 503 as a transient error and retries on its own, so clients drain back in as capacity frees.

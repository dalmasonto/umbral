**Umbra** is a batteries-included web framework for Rust, shaped like Django.
You declare your data, and you get migrations, CRUD, an admin, and (optionally)
a REST API almost for free — with Rust's compile-time guarantees instead of
runtime hopes.

The name is Latin for *shadow*: umbra is a Django shadow.

<Callout type="info">
Umbra is in the design phase. The specs are being written before the code.
Pages get added here as features ship — for now there is only this one.
</Callout>

## Why Umbra exists

Web development in Rust is *unassembled*. Unlike Django or Rails, there is no
mainstream "just works" path: every team picks and wires a web framework, a
database layer, a migration tool, serialization, auth, and background jobs.
Two costs follow:

- **Greenfield productivity** — a basic CRUD/REST API takes days of plumbing.
- **Porting** — moving an existing API to Rust means re-deriving the schema,
  the models, and the conventions by hand. There is no `inspectdb`, no managed
  migrations, no obvious on-ramp.

Umbra targets that on-ramp.

## What you write

A model is a struct. A migration is generated. A handler reads ambient data.
This is the day-to-day feel — not yet working code, but the target shape:

```rust
use umbra::prelude::*;

#[derive(Model)]
pub struct Post {
    pub id: i64,
    pub title: String,
    pub body: String,
    pub published: Option<DateTime<Utc>>,
}

async fn list_published() -> Result<Json<Vec<Post>>> {
    let posts = Post::objects()
        .filter(post::published.is_not_null())
        .order_by(post::published.desc())
        .limit(20)
        .fetch()
        .await?;
    Ok(Json(posts))
}
```

No `State<DbPool>` in the handler signature. No `axum::` types. The ORM is
ambient; the request body, params, and session are explicit arguments.

## Status

Umbra is **pre-alpha**. There is no published crate yet. The architecture,
PRD, and per-subsystem specs are being written first; implementation follows
milestone by milestone. As features land, each gets a doc page here.

If you want the long version, read the design specs in the repository root:
`arch.md` (architecture), `umbra-PRD.md` (product requirements), and
`docs/specs/` (per-subsystem deep specs).

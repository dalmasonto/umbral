Every plugin can ship static assets (CSS, JS, fonts, an SPA bundle). umbra serves them all under one configurable base — `STATIC_URL` (default `/static/`) — so a request for `/static/admin/admin.css` resolves the same way whether the bytes are baked into the binary, read live off disk in development, or fronted by a CDN in production. This is Django's `staticfiles` + `collectstatic` model.

## The two serving modes

A plugin picks the mode that fits its assets — both coexist behind the same URL:

- **Embedded** (zero-config, single binary). The plugin returns the bytes from `Plugin::static_files()`; they're `include_bytes!`-baked and mounted as routes. Best for small, rarely-changed assets. The built-in **admin** uses this — `cargo run` just works, no extra step.
- **Filesystem** (drop-in, CDN-ready). The plugin declares a source directory with `Plugin::static_dirs()`. In development the handler serves files **live off disk** — rebuild your frontend, drop the new files in, and they're served on the next request with no recompile. In production you run `collectstatic` to gather every plugin's assets into one `static_root/` that the framework (or nginx/a CDN) serves. The built-in **playground** uses this for its Vite bundle.

<Callout type="info">
Dev serves from each plugin's source dir; production serves from the collected `static_root`. Same `{{ static(...) }}` URL, two backends — exactly like Django's dev server vs. `STATIC_ROOT`.
</Callout>

## Declaring assets in a plugin

A plugin points `static_dirs()` at an on-disk directory and gives it a unique namespace (the URL/dir segment that prevents collisions between plugins):

```rust
use umbra::prelude::*;
use std::path::PathBuf;

impl Plugin for MyPlugin {
    fn static_dirs(&self) -> Vec<StaticDir> {
        // `dist/assets/app.js` becomes reachable at `/static/myplugin/assets/app.js`
        vec![StaticDir::new(
            "myplugin",
            PathBuf::from(env!("CARGO_MANIFEST_DIR")).join("dist"),
        )]
    }
}
```

A file's path *relative to* `source_dir` is preserved under `<STATIC_URL><namespace>/…`, so `dist/assets/app.js` → `/static/myplugin/assets/app.js`.

## Referencing assets in templates

Use the `static()` global so URLs follow `STATIC_URL` wherever it's configured:

```jinja
<link rel="stylesheet" href="{{ static('myplugin/app.css') }}">
<script src="{{ static('myplugin/assets/app.js') }}"></script>
```

## Configuration

Two settings (overridable via `umbra.toml` or env):

| Setting | Env | Default | Purpose |
|---|---|---|---|
| `static_url` | `UMBRA_STATIC_URL` | `/static/` | URL base every asset is served under. May be an absolute CDN origin (`https://cdn.example.com/s/`). |
| `static_root` | `UMBRA_STATIC_ROOT` | `staticfiles/` | On-disk directory `collectstatic` writes to and production serves from. |

## Using StaticPlugin

`StaticPlugin` (the `umbra-static` crate) is how an app serves its own filesystem assets — and it's also what provides the `collectstatic` command. Register it in the builder:

```rust
use umbra::prelude::*;
use umbra_static::StaticPlugin;

let app = App::builder()
    // Mounted AT `static_url`: the framework's unified pipeline owns
    // `/static/`, and this plugin's directory is served at the bare
    // `/static/<file>` root (your site CSS / images) alongside every
    // plugin's namespaced assets.
    .plugin(StaticPlugin::new("/static", "./static"))
    // Mounted elsewhere (e.g. `/media`): nests independently as its own
    // route, separate from the pipeline.
    .plugin(StaticPlugin::new("/media", "./media"))
    .build()?;
```

A `StaticPlugin` mounted at `static_url` contributes its directory through `static_root_dirs()` so the framework serves `/static/<file>` from it (your project's own CSS / images live at the bare `/static/...` space). Mounted at any other path, it nests its own route independently.

## collectstatic

`collectstatic` is **provided by `StaticPlugin`**, not a core built-in — it's available **only when you register `StaticPlugin`**. A static-free app never sees the command, matching umbra's thin-core, plugin-heavy rule: serving static assets is a plugin capability, so collecting them is too.

Before deploying, gather assets into `static_root/`:

```bash
cargo run -- collectstatic          # copy every plugin's assets into static_root/
cargo run -- collectstatic --clear  # wipe static_root/ first, then collect
```

It collects both kinds of contribution:

- every plugin's **namespaced** `static_dirs()` → `static_root/<namespace>/…`
- every plugin's **site** `static_root_dirs()` (e.g. a `StaticPlugin` at `static_url`) → the `static_root/` root

The result is a complete, CDN-servable tree. Point a CDN or nginx at `static_root/` and set `static_url` to its origin, or let the framework serve it.

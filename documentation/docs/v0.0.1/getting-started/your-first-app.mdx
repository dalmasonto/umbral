The minimal umbra app boots through one builder, registers a database pool, and serves a hand-written route. From there you grow into models, migrations, and plugins.

## The shape

```rust
use umbra::prelude::*;

#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error>> {
    let settings = umbra::Settings::from_env()?;
    let pool = umbra::db::connect(&settings.database_url).await?;

    let app = App::builder()
        .settings(settings)
        .database("default", pool)
        .routes(Routes::new().get("/", || async { "hello, umbra" }))
        .build()?;

    app.serve("127.0.0.1:8000".parse()?).await?;
    Ok(())
}
```

`App::builder().build()` runs five phases under the hood: collect, detect the backend, publish the ambient state, run system checks, and merge plugin routers. From the caller's side it's a single line.

## Settings

`Settings::from_env()` layers defaults, an optional `umbra.toml`, and `UMBRA_`-prefixed environment variables (last wins). The fields that matter on day one:

<CardGrid cols={2}>
  <Card icon="database" title="database_url" description="The DB connection string. Default sqlite::memory:, fine for tests but not for the management CLI. Override with UMBRA_DATABASE_URL." />
  <Card icon="key" title="secret_key" description="Set via UMBRA_SECRET_KEY. The dev default is rejected in production." />
</CardGrid>

<Callout type="warning">
The `sqlite::memory:` default doesn't persist between subcommand invocations, so `cargo run -- migrate` quietly targets a throwaway database. Export `UMBRA_DATABASE_URL=sqlite://app.db?mode=rwc` for any real work.
</Callout>

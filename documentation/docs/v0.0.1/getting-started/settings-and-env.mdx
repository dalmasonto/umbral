Every umbra app has one ambient `Settings` value loaded once at boot from configuration files and environment variables, with later sources overriding earlier ones:

1. **Built-in defaults**: `database_url = "sqlite::memory:"`, `bind_addr = "127.0.0.1:8000"`, etc.
2. **`umbra.toml`** at the project root.
3. **`.env`** at the project root, merged like environment variables for local development.
4. **`UMBRA_`-prefixed environment variables already set in the process.**

After `App::build` runs, `umbra::settings::get()` returns the resolved value from anywhere in the app: handlers, plugin code, task bodies, signal handlers. No state plumbing, no extractors.

## The known fields

```rust
pub struct Settings {
    pub database_url: String,                 // "sqlite::memory:" by default
    pub databases: HashMap<String, String>,   // named secondary pools (optional)
    pub db_max_connections: u32,              // 10 (Postgres pool size)
    pub db_acquire_timeout_secs: u64,         // 30 (Postgres acquire timeout)
    pub secret_key: String,                   // insecure dev default; required for prod
    pub environment: Environment,             // Dev | Test | Prod (default Dev)
    pub allowed_hosts: Vec<String>,           // ["localhost", "127.0.0.1"] by default
    pub log_level: String,                    // "info"
    pub bind_addr: String,                    // "127.0.0.1:8000"
    pub time_zone: Option<String>,            // None (UTC everywhere) by default
    pub static_url: String,                   // "/static/"
    pub static_root: String,                  // "staticfiles/"
    pub extra: HashMap<String, toml::Value>,  // catch-all for app-defined keys
}
```

| Field | Env var | umbra.toml key | Default |
|---|---|---|---|
| `database_url` | `UMBRA_DATABASE_URL` | `database_url` | `"sqlite::memory:"` |
| `databases` | `UMBRA_DATABASES__<NAME>` (double underscore) | `[databases]` table | empty map |
| `db_max_connections` | `UMBRA_DB_MAX_CONNECTIONS` | `db_max_connections` | `10` |
| `db_acquire_timeout_secs` | `UMBRA_DB_ACQUIRE_TIMEOUT_SECS` | `db_acquire_timeout_secs` | `30` |
| `secret_key` | `UMBRA_SECRET_KEY` | `secret_key` | `"umbra-insecure-dev-key-change-me"` (rejected in prod) |
| `environment` | `UMBRA_ENVIRONMENT` | `environment` | `"dev"` |
| `allowed_hosts` | `UMBRA_ALLOWED_HOSTS` (comma-list) | `allowed_hosts` | `["localhost", "127.0.0.1"]` |
| `log_level` | `UMBRA_LOG_LEVEL` | `log_level` | `"info"` |
| `bind_addr` | `UMBRA_BIND_ADDR` | `bind_addr` | `"127.0.0.1:8000"` |
| `time_zone` | `UMBRA_TIME_ZONE` | `time_zone` | `None` (treats naive datetimes as UTC) |
| `static_url` | `UMBRA_STATIC_URL` | `static_url` | `"/static/"` |
| `static_root` | `UMBRA_STATIC_ROOT` | `static_root` | `"staticfiles/"` |

## umbra.toml

The closest analogue to Django's `settings.py`. Lives at the project root next to `Cargo.toml`:

```toml
# umbra.toml
database_url = "sqlite://app.db?mode=rwc"
bind_addr = "0.0.0.0:8000"
secret_key = "replace-me-with-openssl-rand-hex-32"
environment = "prod"
log_level = "info"
allowed_hosts = ["example.com", "www.example.com"]

# Named secondary pools, used by `.on(&pool)` or `Manager::on(pool)`
[databases]
analytics = "postgres://analytics:pw@db-host/analytics"
cache = "sqlite://cache.db?mode=rwc"
```

Commit `umbra.toml` to git for everything that isn't a secret. Keep secrets (`secret_key`, real `database_url` with passwords) out of the committed copy and supply them via env vars instead.

## Environment variables (production / containers)

```bash
export UMBRA_DATABASE_URL="postgres://app:pw@db/app"
export UMBRA_BIND_ADDR="0.0.0.0:8080"
export UMBRA_SECRET_KEY="$(openssl rand -hex 32)"
export UMBRA_ENVIRONMENT="prod"

cargo run -- serve
```

The `UMBRA_` prefix marks the env var as framework-owned. Anything else in your shell is invisible to umbra's settings loader, so there's no accidental capture.

## Reading settings from anywhere

```rust
// In a handler:
async fn home() -> Html<String> {
    let settings = umbra::settings::get();
    Html(format!("running in {:?} mode", settings.environment))
}

// In a Plugin::on_ready:
fn on_ready(&self, _ctx: &AppContext) -> Result<(), PluginError> {
    let settings = umbra::settings::get();
    if matches!(settings.environment, umbra::Environment::Prod) {
        // production-only setup
    }
    Ok(())
}

// In a #[task] body:
#[task]
async fn ping_external_api(payload: PingPayload) -> Result<(), String> {
    let api_key = umbra::settings::get_opt()
        .and_then(|s| s.extra_str("openai_api_key").map(str::to_string))
        .ok_or("OPENAI_API_KEY not configured")?;
    // ... call the API
    Ok(())
}
```

`umbra::settings::get()` returns `&'static Settings` and panics if `App::build` hasn't run. `umbra::settings::get_opt()` returns `Option<&'static Settings>`; use it in code that may run before `App::build` finishes (rare, but possible in macro-emitted helpers, test fixtures, etc.).

<Callout type="info">
The `&'static` lifetime is real: the settings live for the process. You can stash field clones (`settings.secret_key.clone()`) freely; the underlying struct doesn't move.
</Callout>

## Custom settings (the `extra` field)

Most real apps need keys umbra doesn't know about: `OPENAI_API_KEY`, `STRIPE_SECRET`, third-party plugin config. The `extra: HashMap<String, toml::Value>` field catches everything that doesn't match a known field name:

```bash
# Env var
export UMBRA_OPENAI_API_KEY="sk-test"
```

```toml
# umbra.toml - both work
openai_api_key = "sk-test"

# Or nested
[external.openai]
api_key = "sk-test"
model = "gpt-4o"
```

Read with `extra_str` for the common scalar-string case:

```rust
let key = umbra::settings::get()
    .extra_str("openai_api_key")
    .ok_or("missing OPENAI_API_KEY")?;
```

For nested values (`[external.openai]`), index `extra` directly:

```rust
let cfg = umbra::settings::get();
let model: &str = cfg.extra
    .get("external")
    .and_then(|v| v.get("openai"))
    .and_then(|v| v.get("model"))
    .and_then(|v| v.as_str())
    .unwrap_or("gpt-4o");
```

## .env files

`Settings::from_env()` automatically reads a project-root `.env` file before reading `UMBRA_` variables from the process environment. This is the local-dev place for values you do not want to commit:

```bash
cat > .env <<'EOF'
UMBRA_DATABASE_URL=postgres://app:pw@localhost/app_dev
UMBRA_SECRET_KEY=dev-only-secret
EOF

cargo run -- serve
```

Values already exported by the shell, Docker, systemd, or a hosting provider win over `.env`. That keeps production configuration explicit while still making local development convenient.

## Precedence summary

Later wins:

1. Struct defaults (in `Settings::default_*` functions).
2. `umbra.toml` at the project root.
3. `UMBRA_` variables from `.env`, unless the process already set the same key.
4. `UMBRA_`-prefixed env vars already present in the process environment.
5. The `--addr` flag on `umbra serve` (one-off override; doesn't write back to settings).

When the server starts it logs the bound address:

```
INFO umbra_core::app: umbra serving on 127.0.0.1:8000
```

## Loading settings yourself (rare)

`App::builder().settings(...)` accepts a pre-built `Settings`. The common shape uses `from_env`:

```rust
let settings = umbra::Settings::from_env()?;  // figment-loaded
let app = App::builder().settings(settings).build()?;
```

For tests, build a `Settings` literal directly:

```rust
let settings = umbra::Settings {
    database_url: "sqlite::memory:".into(),
    environment: umbra::Environment::Test,
    ..umbra::Settings::from_env()?
};
```

## Purpose

The user-facing query API. `QuerySet<T>` is a lazy, chainable builder whose terminal methods drive sea-query → sqlx → a real Postgres or SQLite query. `T::objects()` is the entry point.

This spec defines the call-site shape (filter/exclude/order/limit, the F/Q analog, aggregates, relation loading, transactions, the raw-SQL escape hatch) and the resolution model (ambient pool, explicit pool, alias). Field-side types (`post::title`, `post::published_at`, etc.) are defined in `04-orm-model-and-fields.md`; this spec uses them but doesn't own them.

At M1 the implementation lands by hand for one hard-coded model (`Post` from the canonical example app), without macros. M3 generates the same shape from `#[derive(Model)]`.

## Concepts

### Manager and entry point

A `Manager<T>` is the value `T::objects()` returns. It's a thin wrapper around a freshly-constructed `QuerySet<T>` and the place where pool selection happens.

```rust
pub struct Manager<T: Model> { /* private */ }

impl<T: Model> T {
    pub fn objects() -> Manager<T> { Manager::default() }
}

impl<T: Model> Manager<T> {
    pub fn using(self, alias: &str) -> Self;       // pick a non-default database
    pub fn on(self, pool: &DbPool) -> Self;        // explicit pool, for tests
    // All QuerySet methods are re-exposed here so the user can write
    // T::objects().filter(...).fetch() without a separate .query() hop.
}
```

The user never constructs a `Manager` directly. `T::objects()` is the only door.

### QuerySet

```rust
pub struct QuerySet<T: Model> { /* private */ }

impl<T: Model> QuerySet<T> {
    // chainable, returns Self
    pub fn filter(self, p: impl Into<Predicate<T>>) -> Self;
    pub fn exclude(self, p: impl Into<Predicate<T>>) -> Self;
    pub fn order_by(self, o: impl Into<OrderExpr<T>>) -> Self;
    pub fn limit(self, n: u64) -> Self;
    pub fn offset(self, n: u64) -> Self;
    pub fn distinct(self) -> Self;

    // shaping (returns a different type)
    pub fn values<C: ColumnTuple<T>>(self, cols: C) -> ValuesQuery<C::Output>;
    pub fn annotate<E: ExprNamed>(self, e: E) -> AnnotatedQuerySet<T, E>;

    // relation loading
    pub fn with<R: Relation>(self, r: R) -> Self;          // JOIN (select_related)
    pub fn with_many<R: Relation>(self, r: R) -> Self;     // separate query, batched (prefetch_related)

    // terminals
    pub async fn fetch(self) -> Result<Vec<T>>;
    pub async fn first(self) -> Result<Option<T>>;
    pub async fn get(self) -> Result<T>;            // errors on 0 or >1 rows
    pub async fn count(self) -> Result<i64>;
    pub async fn exists(self) -> Result<bool>;
    pub async fn delete(self) -> Result<u64>;       // returns affected rows
    pub async fn update(self, set: impl Into<UpdateSet<T>>) -> Result<u64>;

    // aggregate (returns a terminal future)
    pub async fn aggregate<A: Aggregate>(self, a: A) -> Result<A::Output>;
}
```

`QuerySet<T>` is lazy: it carries a sea-query `SelectStatement` plus pool resolution state, and only executes when a terminal method is awaited.

### Predicates and field references

Each model's derive generates a sibling module of column constants:

```rust
// generated by #[derive(Model)] on Post; hand-written at M1
pub mod post {
    pub const id: IntCol<Post> = IntCol::new("id");
    pub const title: StrCol<Post> = StrCol::new("title");
    pub const author_id: IntCol<Post> = IntCol::new("author_id");
    pub const published_at: NullableDateTimeCol<Post> = NullableDateTimeCol::new("published_at");
}
```

The column types (`IntCol`, `StrCol`, `NullableDateTimeCol`, etc.) carry inherent methods that build predicates:

```rust
post::title.eq("rust")                    // = 'rust'
post::title.icontains("rust")             // ILIKE '%rust%'
post::id.in_(&[1, 2, 3])                  // IN (1, 2, 3)
post::published_at.is_not_null()          // IS NOT NULL
post::published_at.before(some_dt)        // < some_dt
```

Predicates compose with `&` (AND) and `|` (OR):

```rust
Post::objects()
    .filter(
        (post::title.icontains("rust") | post::title.icontains("postgres"))
        & post::published_at.is_not_null()
    )
    .order_by(post::published_at.desc())
    .limit(20)
    .fetch()
    .await?
```

This is umbral's predicate composition: boolean predicates combined with operator overloads. F-expressions (referring to one column from another column on the right-hand side) reuse the same column constants: `salary.lt(employee::budget)` works because the column constants implement `Into<Expr<...>>`.

### `.using(alias)` and `.on(pool)` (pool resolution)

```rust
Post::objects().fetch().await?                                   // ambient default pool
Post::objects().using("replica").fetch().await?                  // named pool from App::builder()
Post::objects().on(&test_pool).fetch().await?                    // explicit pool, for tests
```

The resolution rules:

1. `.on(&pool)` wins if set. Used by tests that drive the ORM directly.
2. Otherwise, `.using("alias")` looks up `umbral::db::pool_for(alias)` and uses it. Errors at boot's system check if the alias wasn't registered in `App::builder()`.
3. Otherwise, the ambient pool resolution from `01-app-and-settings.md` applies: the `tokio::task_local!` test override is checked first, then the `umbral::db` `OnceLock`.

`.using()` and `.on()` are inherent on `Manager<T>` (set early) and also on `QuerySet<T>` (set anywhere in the chain). Setting both is a `BuildError`-style panic at terminal time, not a silent override.

## API-shape sketch

A few small slices, against the canonical example app:

```rust
use umbral::prelude::*;

// list the 20 most recent published posts
let recent = Post::objects()
    .filter(post::published_at.is_not_null())
    .order_by(post::published_at.desc())
    .limit(20)
    .fetch().await?;

// fetch one or error
let p = Post::objects().filter(post::id.eq(42)).get().await?;

// count posts per author (aggregate)
let counts: Vec<(i64, i64)> = Post::objects()
    .values((post::author_id,))
    .annotate(count(post::id).as_("post_count"))
    .fetch().await?;

// load posts with their author in one query (JOIN / select_related)
let posts_with_author: Vec<(Post, Author)> = Post::objects()
    .with(post::author)
    .fetch().await?;

// load posts with all their tags (batched second query / prefetch_related)
let posts_with_tags: Vec<(Post, Vec<Tag>)> = Post::objects()
    .with_many(post::tags)
    .fetch().await?;
```

The terminal types (`Vec<(Post, Author)>`, `Vec<(Post, Vec<Tag>)>`) are inferred from the relation arguments. The shape is built by the `Relation` trait's associated types; `04-orm-model-and-fields.md` owns the relation definitions.

## Mechanics and invariants

### Lazy evaluation

`QuerySet<T>` holds a sea-query `SelectStatement`, a pool resolution state, and nothing else. Nothing is sent to the database until a terminal is awaited. Cloning a `QuerySet` is cheap (the `SelectStatement` clones in O(query size)) — a query stub kept around for reuse is fine.

### Transactions

```rust
Db::tx(|tx| async move {
    let post = Post::objects().on(tx).get_or_create(NewPost { ... }).await?;
    PostTag::objects().on(tx).bulk_create(post.id, &tag_ids).await?;
    Ok(())
}).await?
```

`Db::tx` opens a transaction, hands the `&Tx` into the closure, commits on `Ok`, rolls back on `Err`. `.on(tx)` is the parallel form of `.on(&pool)` that accepts a transaction handle. This is the only spot in the spec where the user sees a transaction value; it stays request-scoped per `arch.md §2.2`.

Nested `Db::tx` uses a savepoint. Postgres-only behaviors stay Postgres-only.

### Raw SQL escape hatch

```rust
let rows: Vec<Post> = umbral::db::query_as!(Post, "SELECT * FROM post WHERE created_at > $1", since)
    .fetch_all().await?;
```

`umbral::db::query!` and `umbral::db::query_as!` are re-exports of `sqlx::query!` / `sqlx::query_as!`. They run against the ambient pool unless the user threads one explicitly. Compile-time column checking still applies (`arch.md §7`), which is the whole reason this hatch is sqlx-shaped rather than a stringly-typed `execute_raw`.

### Errors

Every terminal returns `Result<_, umbral::Error>`. `umbral::Error` carries a `From<sqlx::Error>` so `?` flows naturally. The error type is the same across the QuerySet API; no terminal returns a method-specific error type. The `get()` terminal's "not found" and "more than one" cases surface as `Error::NotFound { .. }` and `Error::TooMany { .. }` variants.

## Trade-offs and alternatives considered

**Field constants as a sibling module, vs a `field__lookup` string scheme, vs closure-based.** A `field__lookup` string scheme (passing `"title__icontains"` as a string) leans on runtime reflection and gives up compile-time checking. Closures (`filter(|p| p.title.eq("rust"))`) require the closure parameter to enumerate every field. Sibling modules with typed column constants give type safety, autocomplete, and a concise call site without parsing strings or instantiating dummies. The cost is generating a module per model, which `#[derive(Model)]` handles at M3.

**Predicate composition with `&` and `|`, vs a `q!()` macro.** `&` and `|` lean on `BitAnd`/`BitOr` operator overloads, which gives a syntax close to logical AND/OR without a macro. A `q!()` macro could give a more macro-driven composition syntax (`q!(title == "rust" | title == "postgres")`), but operator overloading on column-built predicates reads naturally to Rust developers and saves a macro. Revisit if proc-macros for complex expressions (subqueries, EXISTS) prove easier inside a `q!()`.

**`.values((col1, col2)).annotate(...)` vs returning `HashMap<String, Value>`.** Typed projection is more verbose at the call site but the result type is `(i64, i64)` not `HashMap<String, Value>`. The Rust-shape win is worth the verbosity; users who genuinely want dynamic results drop down to the raw-SQL hatch.

**Lazy evaluation that holds a `SelectStatement`, vs evaluating on every method.** A lazy query builder makes chaining cheap. The lazy form makes `.clone()` cheap and the call site free of `.await` until the user wants results. The cost is reasoning about when the SQL gets emitted; the rule is "exactly when a terminal is awaited."

## Open questions

- **F-expression syntax for arithmetic.** `salary.lt(employee::budget)` works for direct column references. `salary.lt(employee::budget * 0.9)` needs an expression DSL on the right-hand side. Sketched but unspecified; resolve when a real use case lands (likely M9 or later).
- **Subqueries and `EXISTS`.** Sea-query supports both; the umbral API doesn't yet sketch them. Resolve when a real workload needs subquery composition. Likely a method form (`.subquery_in(field, other_qs)`) plus an `EXISTS` predicate.
- **`bulk_create` / `bulk_update` semantics under Postgres `RETURNING`.** Worth designing once `04-orm-model-and-fields.md` pins down what an insert returns (an Id, the full row, or nothing).
- **Cursor-style streaming reads** for very large result sets. Defer until a workload genuinely needs `.fetch_stream()`; the current API assumes the result fits in memory.

# Media + S3-compatible storage

Date: 2026-06-02
Status: v0 shipped (filesystem). The backend trait shipped as `umbra_core::storage::Storage` (Wave 1, 2026-06-12) — `FsStorage` impl in `umbra-media`, registered ambiently. S3 backend, image library, and signed URLs still deferred to v0.1+. See "Update: v0.1 trait landed" at the end.

## Where this lands today

`umbra-static` already serves a filesystem directory of developer-shipped assets at a chosen URL prefix. v0 of `umbra-media` is the user-upload counterpart: same `tower-http::ServeDir` underneath, but with a `save(filename, content_type, bytes)` helper that writes the file and inserts a row in the framework-tracked `media_file` model.

```rust
App::builder()
    .plugin(MediaPlugin::new("/media", "./media").max_size(10 * 1024 * 1024))
    .build()?;

// Inside an upload handler:
let outcome = media.save(&filename, &content_type, &bytes).await?;
let public_url = outcome.url;
```

The `media_file` table is registered through `Plugin::models()`, so the admin lists it like any other model and the migration engine creates / migrates it without special-case wiring.

## What's missing

Three things gaps.md #49 calls for that v0 deliberately does not ship:

1. **S3-compatible backend.** Object storage instead of local FS.
2. **Image library.** Thumbnail generation, EXIF stripping, format probing.
3. **Signed URLs.** Auth-required reads against the S3 backend.

Each is non-trivial and warrants its own session. The design below lays out how they slot in without breaking the v0 surface.

## Proposed v0.1: extract a `MediaBackend` trait

Today the plugin owns the filesystem code directly. v0.1 extracts a small trait:

```rust
pub trait MediaBackend: Send + Sync {
    async fn save(&self, key: &str, bytes: &[u8]) -> Result<(), MediaError>;
    async fn read(&self, key: &str) -> Result<Vec<u8>, MediaError>;
    async fn delete(&self, key: &str) -> Result<(), MediaError>;
    fn public_url(&self, key: &str) -> Option<String>;
    fn signed_url(&self, key: &str, ttl: Duration) -> Result<String, MediaError>;
}
```

Two impls land alongside it:

- `FsBackend` — what we have today, factored out.
- `S3Backend` — wraps the `aws-sdk-s3` crate. Accepts any S3-compatible endpoint (real S3, MinIO, R2, B2), so the public README doesn't lock users to AWS.

`MediaPlugin::new(mount, FsBackend::new("./media"))` becomes the default; `MediaPlugin::new(mount, S3Backend::new(config))` is the opt-in. The save / list / delete admin paths stay unchanged — they route through the trait.

## Proposed v0.2: image library

A new optional cargo feature `images` gates `image` + `kamadak-exif` deps. When enabled, `MediaPlugin` learns three hooks:

1. **Detect:** on `save`, sniff the first bytes via `image::guess_format` to confirm the declared `content_type`. Mismatched type → reject (configurable).
2. **Thumbnails:** for every image, write `<key>.thumb.<ext>` alongside the original. Sizes configurable; defaults to `[150, 600, 1200]`-wide variants.
3. **EXIF strip:** rewrite the saved file with EXIF removed when the user opts in via `MediaPlugin::strip_exif(true)`. Prevents leaking GPS / camera serials.

The `media_file` row grows two columns: `thumbnails: Json<Vec<ThumbVariant>>` and `image_width / image_height / exif: Option<Json<...>>`. All `#[umbra(noedit)]`, populated by the plugin.

## Proposed v0.3: signed URLs + permissions

Public-by-default media is the right v0 shape; private media needs:

1. `MediaPlugin::require_auth(true)` — every read goes through `require_staff` (or a configurable gate).
2. `media.signed_url(key, ttl)` — returns a presigned S3 URL (for S3) or a JWT-signed cookie path (for FS).
3. Per-row visibility via a `visibility: enum { Public, Private, AuthRequired }` column.

## Out of scope (won't ship under this banner)

- CDN integration. Users can put any CDN in front of `MediaPlugin`'s public URLs; the plugin doesn't need awareness.
- Virus scanning. Outside the framework's value-add — point ClamAV at the directory if you need it.
- Video transcoding. Same answer — pipe to `ffmpeg` outside the request path; record the result via `media.save`.

## Implementation plan when we get back to this

1. Land `MediaBackend` trait + factor `FsBackend` out of v0. No behaviour change.
2. Add `S3Backend` behind a `s3` cargo feature. New dep on `aws-sdk-s3` ~ 2 MB compiled.
3. Land `images` feature with detect + thumb. EXIF strip behind its own flag because users may want EXIF preserved.
4. Land `require_auth` + signed URLs last — they unlock private media but compose naturally on top of the trait.

Each step is one focused PR; v0's surface stays compatible throughout.

## Update: v0.1 trait landed (2026-06-12)

Step 1 of the implementation plan shipped, with one naming change: the backend trait is **`umbra_core::storage::Storage`**, not `MediaBackend`. It lives in `umbra-core` (not the plugin) so the future `FileField` / `ImageField` and the admin can resolve uploads ambiently without depending on `umbra-media` — the dependency-inversion rule from `CLAUDE.md`. The trait surface is:

```rust
#[async_trait]
pub trait Storage: Send + Sync {
    async fn store(&self, filename: &str, content_type: &str, bytes: &[u8])
        -> Result<StoredFile, StorageError>;
    async fn retrieve(&self, key: &str) -> Result<Vec<u8>, StorageError>;
    async fn delete(&self, key: &str) -> Result<(), StorageError>;
    fn url(&self, key: &str) -> String; // public-only; signed URLs deferred
}
```

`StoredFile { key, url }` is the `store` outcome; `StorageError { NotFound, TooLarge, Io, Backend }` carries the failure modes.

**Ambient registration.** A `Storage` backend is registered once at boot and read ambiently — the same "one intentional global" family as the DB pool (`OnceLock`, set-once, first-wins). The accessors are `umbra::storage::{set_storage, storage, storage_opt}`. `MediaPlugin::on_ready` calls `set_storage(self.storage.clone())`, so adding `MediaPlugin` wires the default; an app can also call `set_storage` directly for a custom backend.

**`FsStorage`** is the filesystem impl (the old inline `save` write logic, factored out). `MediaPlugin::new(mount, dir)` constructs one internally (unchanged public API); `MediaPlugin::with_storage(mount, Arc<dyn Storage>)` is the S3-later opt-in. `MediaPlugin::save` now delegates the byte-write to `storage.store(...)` and still inserts the `media_file` row through the ORM.

**Still deferred** (unchanged from above): the `S3Storage` impl, the image library (thumbnails / EXIF / format probing), and signed / auth-gated URLs. The trait is the extraction point those slot in behind.

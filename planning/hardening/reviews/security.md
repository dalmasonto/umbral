# Security review — umbral

Read-only audit, 2026-06-16. Scope: SQL injection, authz, CSRF/headers, secrets/PII, XSS/template-injection, untrusted-input boundaries across `crates/umbral-core/src/` + `plugins/`. Citations are `file:line` against real code. Nothing was modified.

Severity key: **Critical** (exploitable injection / auth-bypass / secret-leak) · **Important** · **Optional** · **FYI** (control correctly in place or non-exploitable note).

---

## Injection (SQL)

The ORM is uniformly parameterized. Every value reaches the DB through sea-query's `SqlxBinder` (`build_sqlx` / `build_with`) or sqlx `bind`/`*_with` — never string-interpolated. Identifiers (table/column) are either compile-time `&'static str` from generated model descriptors, or runtime `String`s from the `ModelMeta` registry, and in all raw paths they are quoted with `"`-doubling before interpolation. I found **no SQL injection**.

- **FYI:** Cross-model search is safe (`crates/umbral-core/src/orm/search.rs:243-278`) — the user `query` binds positionally (`$1` Pg / `?1`,`?2` SQLite), `filter_sql()` is a `&'static str` model-author constant documented as no-user-input (`search.rs:30-38`), `kind` escapes single-quotes (`:117`), every identifier goes through `quote_ident` (`:99-101`, `"`-doubling). SQLite LIKE term is `escape_like`'d and `ESCAPE '\'`-qualified (`:177,229-234`). Correctly in place.
- **FYI:** Dynamic write/read path is safe (`crates/umbral-core/src/orm/dynamic.rs`) — `filter_eq_string` (`:467`), `order_by_col` (`:492`), `fetch_distinct_strings` (`:540`), `search` field-restrict (`:258`) all validate the caller-supplied column against `self.meta.fields` before use; even unvalidated names would be neutralized because every identifier is wrapped in sea-query `Alias::new(...)` (quoted) and every value binds via `build_sqlx`. `junction_table`/`meta.table` come from the model registry, not request data (`:358,515,797`). `search` term binds through `LikeExpr` (`:264`).
- **FYI:** REST django-filter parser is safe (`plugins/umbral-rest/src/filtering.rs:164`) — `field__lookup=value` field name is matched against `col.name` and rejected with `BadInput` if unknown (`:166`); value binds through sea-query; `websearch_to_tsquery` term is a bound `.arg()` not interpolated (`:275-276`); operator `@@` is a hard-coded literal.
- **FYI:** `Expr::cust_with_values` array/json/datetime predicates are safe (`crates/umbral-core/src/orm/column.rs:1813-1857`, `1530-1625`, `1900-2217`) — the SQL template embeds only a `&'static str` column (quoted, `"`-doubled at `:1826`) plus `$1..$n` placeholders; every element binds through the value list. No user value touches the template string.
- **FYI:** `backup.rs` dump/load (`crates/umbral-core/src/backup.rs:247-368`) builds `SELECT/INSERT` with `quoted_ident` (`:375`, `"`-doubling) over `ModelMeta` tables/columns and `?`/`$n` placeholders; values bind via `bind_value`. Unknown columns in a dump fail loudly (`:300,358`). CLI-only, trusted metadata. Safe.
- **FYI:** `apply_slug_from` (`crates/umbral-core/src/orm/write.rs:701`) slugifies a body value into a JSON map (later parameterized) — no SQL construction, no injection.
- **FYI:** Allowed raw-SQL exceptions all check out — `umbral-rls/src/lib.rs:300-335` (RLS `ALTER/CREATE POLICY` DDL, `escape_ident`-quoted, author-trusted policy expressions, Postgres-gated), `umbral-admin/src/models.rs:464,479` (`ensure_tables_for_tests` DDL, documented test exception), `umbral-cache/src/lib.rs:342,381` (`WHERE key = ?` parameterized), `umbral-oauth/src/policy.rs:270,304` (inside `#[cfg(test)]`). gaps ref: NEW (none needed — no defect).
- **Optional:** RLS policy `USING`/`WITH CHECK` expressions are interpolated verbatim from the app author's config (`umbral-rls/src/lib.rs` `render_policy_sql`) — same trust model as `search::filter_sql`. No request data reaches it, but it is author-supplied SQL → fix: keep a doc-comment asserting "constant, never a request value" on the policy builder so the no-user-input contract is explicit. gaps ref: NEW.

## AuthZ / AuthN

- **Important:** Inactive superuser keeps full permission bypass (`plugins/umbral-permissions/src/rest.rs:97-105`, `plugins/umbral-permissions/src/middleware.rs:263-272`) — both the REST `WithPermissions::authenticate` and `is_superuser_safe` load the `AuthUser` row and read `u.is_superuser` but never consult `is_active`. A superuser disabled via `is_active=false` (the natural lockout) still short-circuits every gate to *allow* (`rest.rs:171`, `middleware.rs:243-247`) on any existing session/bearer token. Blast radius is bounded — login/session resolution filters `is_active=true` (`session_user.rs:76`, `login_required.rs:294`), so no *new* session can be minted — but live ones aren't re-gated. Attack → a deactivated/compromised admin retains full authz until their session expires. Fix → read `is_active` in the same lookup and treat super as `is_superuser && is_active`; deny all perms when inactive. gaps ref: NEW.
- **Important:** `AuthUser.password_hash` is serde-`Serialize`d; only the default block-list keeps it off the wire (`plugins/umbral-auth/src/lib.rs:233-234`) — the struct derives `Serialize` with no `#[serde(skip_serializing)]`, and the REST serializer reads DB columns directly via `DynQuerySet`. It does *not* leak by default (`auth_user` is in `DEFAULT_BLOCKED_TABLES`, `plugins/umbral-rest/src/lib.rs:86-97`, 404s), but the documented `.expose(["auth_user"])` opt-in (`lib.rs:509`) without a paired `.hide("password_hash")` dumps argon2 hashes. Attack → operator exposes the user table, forgets to hide the hash → mass hash exfiltration. Fix → add `#[serde(skip_serializing)]` to `password_hash` as defense-in-depth, and/or make `expose("auth_user")` auto-imply hiding it. gaps ref: NEW.
- **FYI:** REST default permission is ReadOnly and writes are actually gated (`plugins/umbral-rest/src/lib.rs:361` `default_permission: Arc::new(ReadOnly)`, resolved by `permission_for` `:287-292`). Every handler calls `cfg.gate(table, action, identity)` after authenticating and before any DB access — list `:1714`, retrieve `:1844`, create `:1873`, update `:2047`, destroy `:2088`, custom `:2143`. `ReadOnly::check` 403s Create/Update/Delete (`permission.rs:177-185` → HTTP 403 `:344`). `create_nested` is only reachable from inside the gated `create`. Anonymous/non-opted-in POST/PUT/PATCH/DELETE → 403, GET → 200. Contract holds.
- **FYI:** Internal-table block-list is broad + enforced at request time (`lib.rs:86-97`): `auth_user`, `session`, `umbral_migrations`, all `permissions_*`, `task_row`, `admin_audit_log`; enforced in both `allow()` (`:984`) and `allowed_model()` (`:1676`), 404. Inbound write bodies strip hidden fields (`strip_hidden_for_write`, `:980`); `apply_overrides` recurses into `?include=` joins so a hidden field can't leak through a relation (`:769-815`). Boot warning for any anonymous + AllowAny table (`:1223-1240`).
- **FYI:** Session identity cannot be forged — random UUIDv4 cookie token (122-bit, `plugins/umbral-sessions/src/lib.rs:221`), DB stores only `SHA-256(token)` (`:197-202`); identity is hash-and-match server-side (`login_required.rs:319-331`, `session_user.rs:64-79`), no header trust. Cookie `HttpOnly; Secure; SameSite=Lax` (`:323-326`, Secure dropped only in Dev). Session-fixation defense destroys the anon session on login (`:498-501`). `is_staff`/`is_superuser` always read fresh from the DB row.
- **FYI:** Extractors behave — `User`/`LoggedIn`/`CurrentIdentity` reject anon with 401 (or configurable 302) (`session_user.rs:224-229`, `login_required.rs:229-233`, `extractors.rs:83-94`); `OptionalUser`/`OptionalIdentity` fail open to `None`, never allow (`session_user.rs:238-242`). Permission query layer fails closed everywhere (`perm.rs:101-104` returns `Ok(false)` on bad codename; `middleware.rs:246` `unwrap_or(false)`; no `unwrap_or(true)`/`Ok(true)` fallback).
- **FYI:** Admin staff-gating is per-handler, not router-level (`plugins/umbral-admin/src/lib.rs:570-722`, guard `auth.rs:33`) — all 33 data/structure routes call `require_staff(...)` first; only login/logout/static-CSS are public. No data route is currently reachable un-gated, but there's no `route_layer`, so a future route that forgets the call would be silently public. Fix (hardening) → wrap the admin router in one staff `route_layer` excluding `/login` + assets. gaps ref: NEW.

## CSRF / Headers

- **Important:** Empty/blank `secret_key` silently "signs" CSRF tokens with an empty HMAC key (`plugins/umbral-security/src/lib.rs:392-394`, `459-460`) — `from_config` treats any `Some(_)`, including `Some("")`, as a resolved secret, and `Hmac::new_from_slice(b"")` accepts a zero-length key. Attack → `UMBRAL_SECRET_KEY=""` makes signed-mode CSRF tokens forgeable (signed under a public empty key) with no warning. Fix → treat empty/whitespace `secret_key` as `None` (degrade-to-plain-double-submit is already the intended fallback) + add a boot `check.rs` finding. The *insecure-default* key is already caught at boot (`crates/umbral-core/src/check.rs:184-214`); the gap is specifically empty-string. gaps ref: NEW.
- **Important:** `csrf_exempt_paths` uses bare `path.starts_with(prefix)` (`plugins/umbral-security/src/lib.rs:407-411`) — exempting `/api` also exempts `/api-internal`, `/apikeys`, and every sub-path. Attack → operator exempts `/api` for bearer REST, but a cookie-authed `/api/account/delete` becomes fully CSRF-exempt. Fix → segment-boundary match (`path == p || path.starts_with(&format!("{p}/"))`). gaps ref: NEW.
- **Important:** Host-header validation is enforced only in `Environment::Prod` (`crates/umbral-core/src/app.rs:1101-1108`, `hosts.rs:80-82`); the Dev default validates nothing. Attack → a deployment that binds `0.0.0.0` but leaves environment Dev gets no Host check → Host-header injection / cache-poisoning / poisoned reset links. Default allowlist is the safe loopback list (`settings.rs:52-54`), the guard is outermost, and the 400 page HTML-escapes the Host (`hosts.rs:97-137`), but there's no warning for non-loopback-bind-in-Dev. Fix → enforce/warn whenever `bind_addr` is non-loopback regardless of declared environment (reuse `is_loopback_bind` from `check.rs:201`). gaps ref: NEW.
- **Important:** HSTS and CSP are opt-in (default `hsts: false`, CSP `None`) (`plugins/umbral-security/src/lib.rs:220,224`) — a prod app mounting `SecurityPlugin` with defaults ships no `Strict-Transport-Security` and no `Content-Security-Policy`, leaving SSL-stripping and giving XSS no CSP backstop. Deliberate dev-safety tradeoff (`:78-87`) but a real prod exposure. Fix → boot warning when `Environment::Prod` + `SecurityPlugin` mounted with `hsts==false` / no CSP. gaps ref: NEW.
- **FYI:** CSRF core is sound — signed double-submit `<random>.<HMAC-SHA256>` with both equality and signature checked (`lib.rs:485-508`); constant-time `subtle::ct_eq` comparison (`:673-676`); GET/HEAD/OPTIONS exempt, POST/PUT/PATCH/DELETE required (`:524-526`); no `X-HTTP-Method-Override`/`_method` smuggling path; empty/missing token rejected (`:429-431,590-624`); JSON endpoints must send `X-CSRF-Token` (not auto-exempt); cookie `SameSite=Lax`, intentionally not HttpOnly (double-submit needs JS read).
- **FYI:** CORS default is strict — empty allowlist, credentials off, no reflect-origin (`crates/umbral-core/src/cors.rs:83-110`); the `*`-origin-with-credentials combo is structurally prevented (`permissive()` forces credentials off `:115-124`; `into_layer()` panics at build if `Any` + credentials `:234-245`). Clickjacking/sniffing headers on by default (`X-Frame-Options: DENY`, `nosniff`, `Referrer-Policy`, COOP) when `SecurityPlugin` is mounted (`lib.rs:208-241`).

## Secrets / PII

- **Important:** argon2 params are implicit `Argon2::default()` (`plugins/umbral-auth/src/lib.rs:531`) — the crate default (Argon2id, m=19456 KiB, t=2, p=1) currently *meets* OWASP, but it's not pinned, so a future `argon2` bump could silently regress cost. Fix → pin `Argon2::new(Algorithm::Argon2id, Version::V0x13, Params::new(19456, 2, 1, None))`. gaps ref: NEW. (Salt is random per password `:530`; verify is constant-time `verify_password` `:543`; unknown-user and wrong-password return identical `InvalidCredentials` `:647-662` — all correct.)
- **Important:** Insecure default `SECRET_KEY` ships and is used as the live CSRF HMAC key in dev (`crates/umbral-core/src/settings.rs:48-50` `"umbral-insecure-dev-key-change-me"`, scaffolded into `umbral.toml` at `umbral-cli/src/scaffold.rs:579`, consumed at `plugins/umbral-security/src/lib.rs:392-393`). Attack → any deployment left on the default has forgeable CSRF tokens. Mitigated: boot hard-fails in `Environment::Prod` (`check.rs:187-195`) and warns on non-loopback bind (`:201-212`). Fix → extend the hard-fail to any non-loopback bind regardless of declared environment (the warning is easy to miss). gaps ref: NEW.
- **Important:** `Masked<T>` `Serialize` emits ciphertext, relying on `.hide()` to keep even the blob out of REST/admin JSON (`crates/umbral-core/src/orm/masked.rs:334-337`) — confidentiality holds (it's ciphertext, not plaintext), and the one built-in consumer (`SocialAccount` tokens) is `#[umbral(noform)]` + admin auto-redacts `secret*`/`password*` (`umbral-admin/src/config.rs:314`), but a custom REST resource exposing a Masked column without `.hide()` dumps the encrypted blob. `Deserialize` treats any inbound string as fresh plaintext (`:345-352`). Fix → document/enforce `.hide()` on exposed Masked columns. gaps ref: NEW.
- **Optional:** `TokenSet` and provider structs derive `Debug` over plaintext tokens / `client_secret` (`plugins/umbral-oauth/src/provider.rs:13`, `providers/google.rs:22`, `providers/github.rs:24`) — no `{:?}` of them is emitted today, but a future `tracing::debug!("{tokens:?}")` or panic backtrace would dump live secrets. Fix → manual redacting `Debug` impl (like `Masked`'s). gaps ref: NEW.
- **Optional:** OAuth token-exchange parse errors interpolate the raw response into `OAuthError::Provider("…: {serde_err}")` (`plugins/umbral-oauth/src/providers/google.rs:67`, `github.rs:65` → surfaced at `routes.rs:229`) — serde's message can include a body fragment; normally the *error* body, low risk. Fix → don't interpolate raw bodies into surfaced/logged errors. gaps ref: NEW.
- **Optional:** No `Masked` key versioning (`masked.rs:98-101`) — rotating `UMBRAL_MASK_PRIVATE_KEY` orphans all existing ciphertext (by-design crypto-shred, but no graceful re-encrypt). Fix → prepend a 1-byte key-id to the sealed blob for a future multi-key keyring. gaps ref: NEW (deferred).
- **FYI:** No secret material is logged anywhere — every `tracing::`/`println!` in oauth/auth/sessions inspected; OAuth logs only provider *keys* and the attacker-supplied `?error=` code, never tokens/PKCE/secrets. `Settings` derives `Debug` but is never debug-printed (no `SECRET_KEY`/`DATABASE_URL` leak).
- **FYI:** `Masked<T>` design is strong — `crypto_box` X25519 + XSalsa20-Poly1305 sealed boxes (authenticated AEAD, non-malleable, `masked.rs:168-170`); fresh ephemeral keypair + random nonce per `seal()`, both prepended to ciphertext (`:139-149`); `Debug`→`"Masked(••••••)"`, `Display`→`"••••••"` (`:295-305`); key must be exactly 32 bytes or `BadKey` (`:180-187`); missing keyring fails serialize loudly, never stores plaintext. Session/bearer tokens stored as SHA-256 of 122/256-bit CSPRNG values (appropriate; not low-entropy passwords).

## XSS / Template injection

- **FYI:** minijinja autoescape ON by default for `.html`/`.htm` (`crates/umbral-core/src/templates.rs:700-706`, `AutoEscape::Html`; only `.txt` is None); loader whitelist kept in sync (`:1023`); admin engine forces HTML autoescape for every template (`plugins/umbral-admin/src/engine.rs:20`). No extension is autoescape-off.
- **FYI:** The `markdown` filter is safe (`templates.rs:337`) — `render_markdown` routes pulldown-cmark output through `ammonia::Builder::clean` (`:585`) *before* `from_safe_string`, so raw-HTML passthrough (`<script>`, `onerror=`, `javascript:` URLs) is stripped by ammonia's default-deny allowlist (only widened by an inert `class` attr `:582-584`). Tested (`:1200-1220`). The `sanitize` filter (`:421`) is `ammonia::clean`; `highlight_css` emits a fixed stylesheet with no user input.
- **FYI:** Admin `tojson` filter is `from_safe_string` into inline `<script>` but escapes `<`/`>`/`&`/U+2028/U+2029 to `\uXXXX`, blocking `</script>` breakout (`plugins/umbral-admin/src/engine.rs:42-59`, tested `:357-381`). Other admin `from_safe_string` sites (`:301-304`) are framework-controlled URL paths, not user input.
- **Optional:** The `img` filter does not validate the URL scheme (`templates.rs:225`) — `{{ user_url | img }}` with `javascript:…` produces `<img src="javascript:…">`. Not a real XSS (modern browsers don't run JS from `<img src>`, and the URL is HTML-escaped so it can't break the attribute or inject `onerror=`), but defense-in-depth. Fix → reject non-`http(s)`/relative schemes in the filter. gaps ref: NEW.

## Input boundaries

- **Important:** OAuth `state` is not single-use / replayable (`plugins/umbral-oauth/src/routes.rs:143` set, `:214` read, never deleted) — the callback validates `state` against the session `FlowState` but never removes it, so a captured `state`+`code` can be replayed until a new flow overwrites the key. Attack → an attacker who obtains a victim's in-flight `state`+`code` replays the callback. Fix → consume-on-read: delete `FLOW_KEY` from the session immediately after the state match, before code exchange. gaps ref: NEW.
- **Important:** No PKCE in the OAuth flow (`plugins/umbral-oauth/src/providers/google.rs:106-119`) — authorize sends no `code_challenge`; there is no `code_verifier` anywhere, and the plugin supports a token-mode SPA (`routes.rs:256-264`). Attack → authorization-code interception for public/SPA clients. Fix → CSPRNG `code_verifier` in session + `code_challenge`/`S256` on authorize + verifier on exchange. gaps ref: NEW.
- **Important:** Upload size cap is enforced only on `MediaPlugin::save`, not the ambient/admin storage path (`plugins/umbral-media/src/lib.rs:426-433` vs `FsStorage::store` `:151-183` which never sees `max_size`) — the admin/form multipart path and any `umbral::storage::storage()` consumer write through `store` with no size check. Attack → bypass the cap, fill disk (DoS). Fix → push the limit into the `Storage` trait / `FsStorage` (or the multipart parser), not just `MediaPlugin::save`. gaps ref: NEW.
- **Optional:** OAuth `state` comparison is plain `!=` (`routes.rs:218`) — server-generated 128-bit UUID vs attacker value, so a timing oracle is impractical, but a constant-time compare matches the framework posture. gaps ref: NEW.
- **Optional:** Return-URL allowlist is a raw prefix match (`plugins/umbral-oauth/src/lib.rs:141-143` `next.starts_with(p)`) — `https://app.example.com` also matches `https://app.example.com.evil.com`; since a login flow mints a bearer token into the return URL fragment (`routes.rs:256-261`), a crafted `?next=` could exfiltrate the token. Fix → match on a parsed origin (scheme+host) or require the prefix to end at a path boundary. gaps ref: NEW.
- **FYI:** Upload filename sanitization is sound (`umbral-media/src/lib.rs:161-175`) — strips `/`,`\`,`\0`, truncates to 120 chars, prefixes a fresh UUIDv4; no path traversal, no overwrite, writes only under `dir.join(key)`. Stored-XSS defense: `neutralise_active_content` (`:127-149`) appends `.txt` to active extensions (`html/svg/js/…`) so ServeDir serves them inert; serve route sets `nosniff` (`:481-485`). Tested (`:576-590`).
- **FYI:** OAuth `state` is a fresh UUIDv4 per flow, stored server-side before redirect, with provider-mismatch rejection (`routes.rs:136,143,211-220`); `?next=` is allowlist-validated and rejects unknown URLs with 400 (`:109-120`).

## Summary

Counts: **0 Critical** · **11 Important** · **8 Optional** · numerous FYI controls-in-place. No SQL injection, no auth bypass for an *active* user, no secret reaches a log, and the core crypto/CSRF/CORS primitives are correctly built. The framework is secure-by-default in the large; the gaps are configuration-conditional or hardening items.

Top 3 exploitable risks:

1. **OAuth: no PKCE + replayable `state`** (`umbral-oauth/src/routes.rs:143-218`, `providers/google.rs:106-119`). Together these expose the authorization-code flow to interception and callback replay, worst for the supported SPA/token mode. Highest real-world exploitability of anything found.
2. **Empty `SECRET_KEY` silently signs CSRF tokens with an empty HMAC key** (`umbral-security/src/lib.rs:392-394,459-460`) — `UMBRAL_SECRET_KEY=""` collapses signed CSRF to forgeable with no warning. Paired with the insecure default key being only *warned* (not hard-failed) on non-loopback dev binds.
3. **`password_hash` is serde-serialized, guarded only by the block-list default** (`umbral-auth/src/lib.rs:233-234`) — one `.expose(["auth_user"])` without `.hide("password_hash")` dumps argon2 hashes. A single config slip leaks credential material.

Runner-up operational risks: inactive-superuser permission bypass on live sessions (`umbral-permissions/src/rest.rs:97-105`), and Host-validation/HSTS/CSP being off by default in non-Prod / opt-in (Host-header injection + SSL-stripping for misconfigured prod deploys).

No code was modified during this review.

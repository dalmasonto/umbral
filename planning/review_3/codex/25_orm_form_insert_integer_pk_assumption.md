# Dynamic Form Inserts Assume Integer Primary Keys

Category: Correctness, Simplicity
Severity: Medium

## Finding

Dynamic form inserts return `i64`. On Postgres, the implementation asks for `RETURNING <pk>` and falls back to `0` when the primary key is not an integer. Admin row creation works around parent primary keys by reading non-integer IDs from the submitted form, but inline child creation still ignores the returned ID and the API contract remains integer-only.

The framework has broader support for string and UUID primary keys elsewhere, so this is a core ORM surface that still carries an old integer-PK assumption.

## Evidence

- `crates/umbral-core/src/orm/dynamic.rs:1399-1408` documents `insert_form` as returning `last_insert_rowid`.
- `crates/umbral-core/src/orm/dynamic.rs:1420-1437` Postgres `insert_form` returns `row.try_get::<i64>(pk).unwrap_or(0)` for the returned primary key.
- `crates/umbral-core/src/orm/dynamic.rs:1539-1544` `insert_form_in_tx` also returns `i64`.
- `crates/umbral-core/src/orm/dynamic.rs:1557-1573` transactional Postgres form insert has the same non-integer fallback.
- `plugins/umbral-admin/src/rows.rs:228-238` works around non-integer parent PKs by returning the form value when the model primary key is not integer-like.
- `plugins/umbral-admin/src/inlines.rs:397-402` inserts new inline child rows and discards the integer return value.

## Risk

Models with UUID or string primary keys can have surprising admin/form behavior:

- If the primary key is generated by the database instead of supplied in the form, the caller receives `0` and cannot reliably link follow-up work to the created row.
- Admin code has to carry local workarounds for parent rows, and those assumptions can diverge from inline or plugin code.
- The ORM API advertises a primary-key result but loses the actual key shape.

## Recommendation

Return a primary-key value that preserves shape, for example `serde_json::Value` or a small `InsertedPk` enum. Keep an integer convenience accessor if needed, but make the core API represent string, UUID, and integer PKs uniformly.

Then update admin row and inline code to consume the typed/dynamic primary-key value rather than reconstructing it from form inputs.

## Suggested Tests

- Dynamic `insert_form` on a UUID primary-key model returns the UUID, not `0`.
- Dynamic `insert_form_in_tx` on a string primary-key model returns the string primary key.
- Admin parent create with generated UUID/string PK can create inline children in the same transaction.
- Integer primary-key behavior remains unchanged.

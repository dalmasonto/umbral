39. **Filtering a `#[umbral(choices)]` field takes a raw string, not the `Choices` variant — the typed field constant loses the enum.** Every filter on a choices-backed column in the codebase passes a string literal: `product::STATUS.eq("active")`, `post::STATUS.eq("published")`, `ful_job::STATUS.eq("pending")` — there are zero `.eq(SomeEnum::Variant)` uses anywhere. So even though `status` is declared as a strongly-typed `Choices` enum on the model, and the field handle `task::STATUS` is a typed column, `.eq()` accepts an arbitrary `&str`. A typo (`"in_progres"`, `"done "`, a value that doesn't match `#[choices(rename_all = ...)]`) compiles fine and silently matches no rows. This directly contradicts the framework's stated value prop — CLAUDE.md lists "stringly typed filters where a typed filter is available" among the boring mistakes the ORM is supposed to prevent, and the whole "declare the enum once, every surface honours it" story leaks here: the write path and admin dropdown are enum-typed, but the query path is not. **Proposed fix:** for a field declared `#[umbral(choices)]` over an enum `E: Choices`, generate the column handle so `.eq(E)` / `.ne(E)` / `.in_([E])` accept the variant and serialize it through the *same* `Choices` mapping used at write time (so `#[choices(rename_all = "snake_case")]` stays single-sourced) — `task::STATUS.eq(TaskStatus::InProgress)` renders `status = 'in_progress'`. Keep a string-accepting form for dynamic/late-bound callers (`DynQuerySet`, admin filters, query-param parsing) where the enum type isn't statically known. Net effect: a bad status becomes a compile error, and the rename mapping can never drift between the write and read paths. Ship with a test: `Task::objects().filter(task::STATUS.eq(TaskStatus::InProgress))` returns the in-progress rows and the generated SQL binds `'in_progress'`, and a round-trip create+filter reads the row back. (surfaced writing the "One Model, Every Surface" tutorial — the natural `task::STATUS.eq(TaskStatus::InProgress)` had to be downgraded to `.eq("in_progress")` and a prose claim about compile-checked status filtering retracted because it wasn't true)

**SHIPPED** (commit `efde7982`, non-breaking variant). The `Choices` derive now emits `impl From<Enum> for String` that serializes through `ChoiceField::as_str` — the exact mapping the write path uses — so the existing `StrCol::eq<S: Into<String>>` / `ne` accept the enum variant directly: `status::STATUS.eq(TaskStatus::InProgress)` compiles and binds `'in_progress'`, single-sourced with `#[choices(rename_all = ...)]`.

Design decision (chosen with the maintainer): the **non-breaking** approach over the enum-only `ChoiceCol<M, E>` that the original proposal implied. Trade-off — the string form (`.eq("in_progress")`) still compiles, so a typo'd string is *not* a compile error (it matches no rows, as before). This kept zero churn: no call sites in the workspace, `examples/shop`, or the live `umbral_website` had to change. Within the workspace, `umbral-tasks.status` is a plain `String` (not choices), so nothing there was affected either way.

Test: `crates/umbral-core/tests/choices_typed_filter.rs` — seeds rows via the typed write path (enum status), then asserts `ticket::STATUS.eq(TicketStatus::InProgress)` selects the right rows, matches the `.eq("in_progress")` string form, `.ne(variant)` works, and a create→filter round-trip reads the enum back. Doc: `documentation/docs/v0.0.1/orm/column-types.mdx` updated to lead with the typed form (was documenting the old string-only limitation).

`.in_([E])` was in the original proposal but `StrCol` has no `in_` method at all (only `IntCol`/`UuidCol` do), so choices-column `in_` remains a separate, pre-existing gap — out of scope here.

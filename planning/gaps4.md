## FRAMEWORK ENHANCEMENTS (surfaced by downstream app work â€” local_task_tracker)

36. [ ] **`GroupPolicy` is synchronous, so realtime rooms can't be gated on async state (DB membership).** `GroupPolicy::can_join(&self, user_id: Option<&str>, group: &str) -> bool` (and `can_send`) is a sync fn with no async variant. A realtime app that wants to admit a user to `project:{id}:*` only if they hold an active membership row must run a DB query at the SSE/WS handshake â€” which cannot be `.await`ed inside a sync `can_join`. This matters for confidentiality, not just metadata: with field-projected exposures (e.g. a chat message exposing `body_markdown`), a non-member who joins the group receives real row content over SSE, and REST-side scoping does not cover the realtime path. Current downstream workaround (local_task_tracker SP-A): `tokio::task::block_in_place` + `Handle::current().block_on(...)` inside `can_join` â€” correct only on a multi-thread runtime (panics on current-thread) and blocks one worker thread per room-join for the duration of the membership query. **Proposed fix:** add an async policy hook to `RealtimePlugin` â€” e.g. an `AsyncGroupPolicy` trait with `async fn can_join(&self, identity: Option<&Identity>, group: &str) -> bool` (and `can_send`), resolved at the handshake alongside the existing async `identity_resolver`; keep the sync `GroupPolicy` for shape-only checks. Downstream can then drop the `block_in_place` bridge. Ship with a test that a non-member is refused a private group at the SSE handshake while a member is admitted, and that the handshake does not block a worker thread. (local_task_tracker phase2 SP-A)

6. [ ] Admin dashboard **widget catalog** (`GET /admin/api/dashboard/widgets` — `handlers::dashboard::dashboard_catalog`) doesn't filter by `widget.permission`. A user without a widget's codename still sees it in the "add widget" catalog, adds it, then gets 403 on the data fetch. UX gap, not a security hole (the data endpoint IS gated — see `dashboard_widget_data`). Fix: reuse `permcheck::has_codename` to drop catalog entries the user can't load. Surfaced by the custom-views final review (features #76).
7. [ ] Custom admin views (`AdminPlugin::view(AdminView)`) don't validate the path at registration. A view whose path collides with a built-in admin route (`login`, `logout`, `upload-image`, `api/palette`, …) or duplicates another view panics inside axum's router at boot with a cryptic message (fails fast, not silently — multi-segment paths like `reports/sales` coexist fine via static-over-param precedence). Fix: reject reserved prefixes / duplicate paths in `AdminView::new` or the mount loop with a clear error. Surfaced by the custom-views final review.
8. [ ] The custom-view / dashboard render filter (`view::accessible_widget_sections_json`) runs `permcheck::has_codename` sequentially per widget. Fine for typical widget counts (5–20); for large sections it adds per-widget DB latency to every page render. Optimize with a batched permission lookup / `join_all` if widget counts grow. Deferred from the widget-permission enforcement work.

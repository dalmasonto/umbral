9. [x] REST nested writes are create-only; PATCH/PUT ignores nested child arrays — shipped

   `RestPlugin` supported writable nested children only on `POST`; the `update` handler was flat and ignored `cfg.nested`, so a PATCH carrying `{ "items": [...] }` handed the array to the ORM as an unknown column instead of upserting children.

   **Shipped:** `update` now splits declared nested arrays out of the body and upserts children on ONE `umbral::db::begin()` tx (parent update + child writes commit/roll-back together). **Reconciliation: upsert, no implicit deletes** — item WITH the child pk → update (scoped to this parent via the FK; a cross-parent pk is a 404); WITHOUT a pk → create. Rows absent from the payload are untouched. Full replace-set (delete-the-missing) stays a future opt-in (`ResourceConfig::nested_sync(...)`). Test: `plugins/umbral-rest/tests/nested_updates.rs`. Superseded/extended by #10 (recursion). The `update_json_in_tx`-return-is-not-affected-count footgun found here is captured in `.claude/skills/dynqueryset-update-return-semantics.md`.
10. [x] Nested writes only went one level deep; grandchildren were silently dropped — shipped

   `create_nested`/`update_nested` iterated only the parent's `.nested()` specs and inserted each child flat, so a level-3 array (e.g. `order.items[].components[]`) rode along inside the child object and — because the dynamic insert path iterates the child table's columns and validation doesn't flag unknown keys (`crates/umbral-core/src/orm/validation.rs:83`) — was **silently dropped**: no error, no rows. Silent data loss, the exact anti-pattern CLAUDE.md's "fix, don't patch" calls out.

   **Shipped:** both writers are now recursive (`insert_nested_tree` / `upsert_nested_child` in `plugins/umbral-rest/src/lib.rs`). Nesting is driven per table from `cfg.nested`, so a subtree is written iff its parent's table *also* declared `.nested(...)` — one level per declaration, arbitrary depth. Each level: FK auto-set from the parent's just-inserted pk (create) or ownership-scoped upsert (update); `MAX_NEST_DEPTH = 16` guards cyclic/self-referential declarations with a 400. Test: `plugins/umbral-rest/tests/nested_deep.rs` (3-level create + deep upsert + depth-3 cross-parent 404 rollback).

   **Follow-up (deferred):** declaring `.nested()` on a mid-level table also exposes it as a routed REST resource. If a caller wants deep nesting *without* exposing the intermediate table, we need a declaration that registers nesting without mounting routes (e.g. `ResourceConfig::for_::<T>().nested_only(...)` or a plugin-level nested-map). Log a new gaps3 entry if/when that's needed.
11. [ ] Auth JSON routes are slash-inconsistent with REST resources → `/api/auth/login/` 404s under the default `SlashRedirect::Append`

   Found building a real consumer backend (web3clubs_fc). `AuthPlugin::with_default_routes()` registers the JSON auth routes WITHOUT a trailing slash — `POST /api/auth/login`, `/register`, `/me`, `/logout` (`plugins/umbral-auth/src/auth_routes.rs:201-214`, `.route(&format!("{prefix}/login"), post(login))`). But `RestPlugin` resources use a TRAILING slash (`/api/fixture/`), and the `startproject` scaffold's `main.rs` turns on `.slash_redirect(SlashRedirect::Append)` by default. Net effect for a consumer: hitting `POST /api/auth/login/` — the natural thing to try, since every REST endpoint ends in `/` — returns **404** (the Append policy would redirect a no-slash request *to* the slash form, but the route registered is the no-slash form, so the slash form matches nothing). The no-slash form works, but the inconsistency is a silent footgun: it cost real debugging time (login appeared broken until the exact path was reverse-engineered), and a 307/308 redirect on POST wouldn't preserve the body anyway.

   **Impact:** every consumer that follows the REST plugin's trailing-slash convention (or whose HTTP client auto-appends) gets a confusing 404 on login/register — the first thing they wire up.

   **Fix options (pick one, be consistent):** (a) register the auth routes WITH a trailing slash to match REST resources (and let Append handle the no-slash form), or (b) register BOTH forms (like REST's collection routes already do — `{base}/{table}/` and `{base}/{table}`), or (c) have `with_default_routes()` read the app's `SlashRedirect` policy and register the matching shape. Option (b) is the most forgiving and mirrors what `RestPlugin::routes()` already does for collection paths. At minimum, document the exact (no-slash) auth paths prominently in `documentation/docs/v0.0.1/auth/` and the scaffold README, since the scaffold ships `SlashRedirect::Append` on by default.
12. [ ] `GET /oauth/{provider}/login` returns 500 (not 404) for a provider key that isn't registered

   Also found in web3clubs_fc. When `OAuthPlugin` has no providers registered (e.g. dev without `UMBRAL_OAUTH_GOOGLE_CLIENT_ID/_SECRET`), `GET /oauth/providers` correctly returns `{"providers":[]}` with 200 — but `GET /oauth/google/login` (a provider key that exists as a concept but isn't configured/registered) returns **500**, not a 404/400. A client-reachable route with a foreseeable input (an unconfigured or misspelled provider key) shouldn't 500. The route handler in `plugins/umbral-oauth/src/routes.rs` should look the provider up and return `404 Not Found` (or `400` with `{"error":"unknown_provider"}`) when the key isn't in the registered set, reserving 500 for genuine internal failures.

   **Impact:** low in practice — the intended pattern is that the SPA reads `/oauth/providers` first and only renders a button for a registered provider, so it never hits `login` for an unconfigured key. But a hand-typed/stale URL, a monitoring probe, or a frontend bug turns a 4xx-shaped condition into a 500 (noisy alerts, misleading logs). Cheap to fix at the lookup site.
13. [ ] SQLite `AlterColumn` (null-flip / type change) fails with `FOREIGN KEY constraint failed` when the table has inbound FKs

   Found in web3clubs_fc. Changing a column that requires SQLite's table-recreation dance — e.g. flipping `Fixture.opponent: String` → `Option<String>` (NOT NULL → nullable) — generates an `AlterColumn` migration that, on apply, dies with `Sqlx(Database(SqliteError { code: 787, message: "FOREIGN KEY constraint failed" }))`. The `fixture` table is referenced by inbound FKs from `attendance`, `payment`, and `chat_message`. SQLite's 12-step ALTER-via-recreate (create new table → copy → drop old → rename) trips referential integrity because the child rows still point at the old table during the swap, and the recreation runs with `foreign_keys` ON.

   The SQLite migration executor (`crates/umbral-core/src/orm/queryset/backend_sqlite.rs` / the migrate apply path) needs to wrap a table-recreation alter in `PRAGMA foreign_keys=OFF; … ; PRAGMA foreign_key_check; PRAGMA foreign_keys=ON;` (the official SQLite recipe — pragmas must be set OUTSIDE the transaction, since `foreign_keys` is a no-op inside one). Without it, **any** `AlterColumn` on a parent table is unappliable on SQLite, which is a real wall for the declare→migrate loop once a schema has relations.

   **Severity:** medium-high — it bricked `cargo run` boot (auto-migrate aborts → server won't serve) until the offending migration was removed. Transactional DDL saved the data (the failed alter rolled back cleanly, table intact), but the loop is dead-ended until this is fixed. **Workaround used:** kept the columns non-null and had the client send non-blank placeholder values. **Also note:** the REST create validator rejects an explicitly-sent blank string on a required Text column (`This field cannot be blank`), so "just send `""`" isn't an escape hatch — hence the placeholder.
14. [ ] `update_or_create`'s UPDATE branch emits `bulk_post_save`, not per-row `post_save` — silently bypasses signal/realtime consumers

   Found wiring realtime notifications in web3clubs_fc. `umbral-realtime`'s `on_model::<T>` bridge (and any `umbral-signals` `on_model().post_save`) subscribes to the per-row `post_save:<table>` signal. `QuerySet::update_or_create` fires that signal on its CREATE branch (it calls `self.create()`, which emits `emit_post_save::<T>(&row, true)` — queryset/mod.rs:4640), but its UPDATE branch (`queryset/mod.rs:4114`, the `do_update!` macro) applies the change via `self.filter(pk_pred).update_values(map)`, which emits `bulk_post_save:<table>` (with ids) — NOT the per-row `post_save`. So a handler registered via `on_model` sees the row's first write (insert) but NEVER its subsequent `update_or_create` upserts. For a realtime "notify on change" use case this means the first event fires and every status change after is silent.

   The same asymmetry exists in `QuerySet::update()` / `update_values()` generally (bulk paths emit bulk signals by design), but `update_or_create` is the surprising one because its CREATE half DOES emit the per-row signal — the two halves of one convergent API behave differently, so which signal fires depends on whether the row already existed. That's very hard to reason about from the call site.

   **Options:** (a) have `update_or_create`'s update branch fetch-and-`save()` the single row (per-row `post_save`) instead of `update_values`, so the whole API is per-row-signal-consistent; or (b) at minimum document loudly that `on_model`/`post_save` won't see `update_or_create` updates and point consumers at `save()` or an explicit push. **Workaround used:** pushed the payment notification explicitly from the handler with `Realtime::to_user(...)` rather than relying on the `on_model` bridge.

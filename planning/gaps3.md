[ ] Flash messages no-op without a pre-existing session, so umbral-auth's form-action endpoints (`with_form_routes`) silently lose the error on an **anonymous first-visit** form failure (e.g. a bad-credentials login from a brand-new visitor with no session cookie yet). `umbral_sessions::messages::Messages::add` (plugins/umbral-sessions/src/lib.rs:1021) returns early when the request carries no session token; umbral sets session cookies *explicitly* (handlers pass response headers to `login_with_request`), not via middleware auto-tracking, so `Messages` — an extractor with no response handle — cannot create one. The "redirect back **with errors**" promise therefore holds for authed users and any session-backed flow (incl. CSRF flows that establish a session), but not for the anonymous cold-start case. Proper fix (Django-style, needs a design call): make a flash write ensure an anonymous session exists and have the response path emit its `Set-Cookie` — either a new `Messages`/sessions primitive that yields the cookie to attach, or middleware that tracks a mid-request session creation and sets the cookie automatically. Discovered in the Task 14 review of the auth form-action surface (feat/auth-full-surface).
